第6章：高级类型 ​

TypeScript 拥有世界级的类型系统，支持强大的类型级编程功能，这些功能可能会让最挑剔的 Haskell 程序员都感到嫉妒。正如你现在所知道的，这个类型系统不仅表达能力极强，而且易于使用，使得声明类型约束和关系变得简单、简洁，并且在大多数情况下是推断出来的。

我们需要如此表达丰富且不寻常的类型系统，因为 JavaScript 是如此动态。建模像原型、动态绑定的 this、函数重载和总是变化的对象这样的东西需要丰富的类型系统和一套类型操作符工具带，这会让蝙蝠侠都要做双重检查。

我将从深入探讨 TypeScript 中的子类型、可赋值性、变异性和扩展开始本章，为你在过去几章中一直在发展的直觉提供更多定义。然后我将介绍 TypeScript 基于控制流的类型检查功能的更多细节，包括细化和完整性，以及带有一些高级类型级编程功能的总体类型：键入和映射对象类型，使用条件类型，定义你自己的类型守卫，以及像类型断言和明确赋值断言这样的逃生舱口。最后，我将介绍从你的类型中挤出更多安全性的高级模式：伴随对象模式，改进元组类型的推断，模拟名义类型，以及安全地扩展原型。

类型之间的关系 ​

让我们从更仔细地观察 TypeScript 中的类型关系开始。

子类型和超类型 ​

我们在"谈论类型"中简单讨论了可赋值性。现在你已经看到了 TypeScript 提供的大部分类型，我们可以更深入地探讨，从顶部开始：什么是子类型？

子类型定义 ​

子类型（SUBTYPE）

如果你有两个类型 A 和 B，并且 B 是 A 的子类型，那么你可以在任何需要 A 的地方安全地使用 B（图 6-1）。

如果你回顾第3章开头的图3-1，你会看到 TypeScript 内置的子类型关系。例如：

  • Array 是 Object 的子类型
  • Tuple 是 Array 的子类型
  • Everything 是 any 的子类型
  • never 是 everything 的子类型
  • 如果你有一个扩展了 Animal 的类 Bird，那么 Bird 是 Animal 的子类型

从我刚才给出的子类型定义，这意味着：

  • 任何需要 Object 的地方你也可以使用 Array
  • 任何需要 Array 的地方你也可以使用 Tuple
  • 任何需要 any 的地方你也可以使用 Object
  • 你可以在任何地方使用 never
  • 任何需要 Animal 的地方你也可以使用 Bird

正如你可能已经猜到的，超类型是子类型的反面。

超类型定义 ​

超类型（SUPERTYPE）

如果你有两个类型 A 和 B，并且 B 是 A 的超类型，那么你可以在任何需要 B 的地方安全地使用 A（图 6-2）。

再次从图3-1的流程图中：

  • Array 是 Tuple 的超类型
  • Object 是 Array 的超类型
  • Any 是 everything 的超类型
  • Never 是 nothing 的超类型
  • Animal 是 Bird 的超类型

这只是子类型工作方式的反面，没有更多内容。

变异性（Variance） ​

对于大多数类型，判断某个类型 A 是否是另一个类型 B 的子类型是相当容易的。例如，简单类型如 string、string | number，以及你可以查看它们的流程图（图3-1），或者通过它推理（number 包含在联合 number | string 中，所以 number 必须是 number | string 的子类型）。

但对于参数化（泛型）类型和其他更复杂的类型，情况变得更加复杂。考虑这些情况：

  • 当 Array<A> 是 Array<B> 的子类型时？
  • 当形状 A 是另一个形状 B 的子类型时？
  • 当函数 (a: A) => B 是另一个函数 (c: C) => D 的子类型时？

子类型规则对于包含其他类型的类型（即，具有类型参数的东西，如 Array<A>、具有字段的形状如 {a: number}，或函数如 (a: A) => B）并不总是清晰的。事实上，子类型规则对于这些复杂类型是一个大的分歧点，几乎没有两种编程语言以完全相同的方式处理它们！

子类型规则对于包含其他类型的类型被称为变异性，这是编程语言中一个重要的不一致来源——几乎没有两种语言以相同的方式处理它们。

为了使以下规则更容易阅读，我将引入一些语法，让我们以简洁和精确的方式谈论 TypeScript。这只是你和我分享通用语言的一种方式，当我们谈论类型时。不要担心，我发誓语法很简单！

  • A <: B 意味着"A 是 B 的子类型或与 B 类型相同"
  • A >: B 意味着"A 是 B 的超类型或与 B 类型相同"

形状和数组变异性 ​

为了避免一些编程语言在复杂类型上的子类型规则分歧，让我通过一个复杂类型的例子来引导你：形状。假设你有一个描述应用程序中用户的形状。你的应用程序有两种类型的用户，看起来像这样：

typescript
// 从服务器获得的现有用户
type ExistingUser = {
  id: number
  name: string
}

// 尚未保存到服务器的新用户
type NewUser = {
  name: string
}

现在假设你公司的实习生负责编写一些删除用户的代码。他们开始是这样的：

typescript
function deleteUser(user: {id: number, name: string}) {
  delete user.id
}

let existingUser: ExistingUser = {
  id: 123456,
  name: 'Ima User'
}

deleteUser(existingUser)

deleteUser 接受一个类型为 {id: number, name: string} 的对象，并传递一个 existingUser，类型为 {id: number, name: string}。注意 id 的类型是期望类型（number | undefined）的子类型，因此整个形状 {id: number, name: string} 是 {id: number, name: string} 的子类型，所以 TypeScript 认为这是可以的。

你看到这里的安全问题吗？这是一个微妙的问题：在将 ExistingUser 传递给 deleteUser 后，TypeScript 不知道用户的 id 已被删除，所以如果我们读取 existingUser.id，删除后用 deleteUser(existingUser)，TypeScript 仍然认为 existingUser.id 是 number 类型！

显然，在一个地方使用超类型可能是不安全的。那么为什么 TypeScript 允许它？一般来说，TypeScript 的设计不是为了完全安全，而是在捕获真正的错误和实际可用性之间取得平衡。所以为什么 TypeScript 允许它？在实践中，删除属性是相对罕见的，TypeScript 是宽松的，让你将对象分配给期望其超类型的地方。

相反方向呢——你能将对象分配给期望其子类型的地方吗？

让我们为传统用户添加一个新类型，然后删除该类型的用户（想象你要添加类型到现有代码库，你开始使用 TypeScript）：

typescript
type LegacyUser = {
  id?: number | string
  name: string
}

let legacyUser: LegacyUser = {
  id: '793331',
  name: 'Xin Yang'
}

deleteUser(legacyUser) // 错误 TS2345: 类型 'LegacyUser' 的参数不能
                       // 分配给类型 '{id: number, name: string}' 的参数。
                       // 未定义，name: string}' 类型 'string' 不能
                       // 分配给类型 'number | undefined'。

当我们传递一个属性类型是期望类型的超类型的形状时，TypeScript 会抱怨。这是因为 id 是 string | number | undefined，而 deleteUser 只处理 id 是 string | number | undefined 的情况。

TypeScript 的行为如下：如果你期望一个形状，你也可以传递一个属性类型是其期望类型的子类型的形状，但你不能传递一个属性类型是其期望类型的超类型的形状。当谈论形状时，我们说 TypeScript 形状（对象和类）在其属性中是协变的。这意味着对于对象 A 可分配给对象 B，A 的每个属性必须 <: B 的相应属性。

更一般地，协变只是四种变异性之一：

不变性（Invariance） ​

你想要确切的 T

协变性（Covariance） ​

你想要 <: T

逆变性（Contravariance） ​

你想要 >:T

双变性（Bivariance） ​

你想要 <: T 或 >: T

我们刚刚看到 TypeScript 对象在其属性中是协变的。让我们看看其他一些类型的变异性行为。

在 TypeScript 中，除了函数参数类型（它们是逆变的）之外，每个复杂类型在其成员中都是协变的——对象、类、数组和函数返回类型——只有一个例外：函数参数类型，它们是逆变的。

注意

并非所有语言都做出相同的设计决定。在某些语言中，对象在其属性类型中是不变的，因为虽然它更安全，但它可能使类型系统变得乏味，因为你最终需要指定自己的变异性。一些语言对可变和不可变对象有不同的规则（尝试自己推理！）。一些语言有不同的规则，用于指定自己类型的变异性。

当设计 TypeScript 时，其作者选择了在易用性和安全性之间取得平衡。当你使对象在其属性类型中不变时，即使它更安全，它也可能使类型系统变得乏味，因为你最终需要删除 deleteUser 中的 id，然后如果我们读取它，它将是完全安全的传递给 deleteUser 的对象，其超类型是预期类型）。

函数变异性 ​

让我们从一些例子开始。

函数 A 是函数 B 的子类型，如果 A 具有相同或更低的参数数量（参数数量）比 B 和：

  1. A 的 this 类型要么未指定，要么 >: B 的 this 类型
  2. A 的每个参数都 >: B 中相应的参数
  3. A 的返回类型 <: B 的返回类型

读几遍，确保你理解每个规则的含义。你可能已经注意到，对于函数 A 是函数 B 的子类型，我们说 A 的参数类型必须在 B 中 >: 1 为什么方向会像那样翻转？为什么不是简单的 <: 对于每个组件（this 类型、参数类型和返回类型），就像它对对象、数组、联合等一样？

为了回答这个问题，让我们深入研究自己。我们将通过定义三种类型开始（我们将使用 class 来清晰，但这适用于任何类型选择，其中 A <: B <: C）：

typescript
class Animal {}
class Bird extends Animal {
  chirp() {}
}
class Crow extends Bird {
  caw() {}
}

在这个例子中，Crow 是 Bird 的子类型，Bird 是 Animal 的子类型。也就是说，Crow <: Bird <: Animal。

现在，让我们定义一个接受 Bird 并使其鸣叫的函数：

typescript
function chirp(bird: Bird): Bird {
  bird.chirp()
  return bird
}

到目前为止，一切都很好。TypeScript 让你传递什么样的东西到 chirp？

typescript
chirp(new Animal) // 错误 TS2345: 类型 'Animal' 的参数不能分配
                  // 给类型 'Bird' 的参数。

chirp(new Bird)   // OK
chirp(new Crow)   // OK

你可以传递 Bird 的实例（因为这就是 chirp 的参数 Bird 的类型）或 Crow 的实例（因为它是 Bird 的子类型）。显然，传递 Animal 的子类型是有效的。

让我们创建一个新函数。这次，它的参数将是一个函数：

typescript
function clone(f: (b: Bird) => Bird): void {
  // ...
}

clone 需要一个函数 f，该函数接受 Bird 并返回 Bird。你可以安全地传递什么类型的函数给 f？显然你可以传递一个接受 Bird 并返回 Bird 的函数：

typescript
function birdToBird(b: Bird): Bird {
  // ...
}
clone(birdToBird) // OK

那么一个接受 Bird 并返回 Crow 或 Animal 的函数呢？

typescript
function birdToCrow(b: Bird): Crow {
  // ...
}
clone(birdToCrow) // OK

那么一个接受 Bird 并返回 Crow 或 Animal 的函数呢？

typescript
function birdToCrow(b: Bird): Crow {
  // ...
}
function birdToAnimal(b: Bird): Animal {
  // ...
}

你可以传递 birdToCrow 给 clone，因为 Crow <: Bird，所以返回 Crow 的函数可以安全地用在期望返回 Bird 的地方。但你不能传递 birdToAnimal，因为 Animal >: Bird，所以返回 Animal 的函数不能安全地用在期望返回 Bird 的地方。

现在让我们尝试一些不同的东西。让我们创建一些函数，它们接受不同类型的参数，但都返回 Bird：

typescript
function animalToBird(a: Animal): Bird {
  // ...
}
function crowToBird(c: Crow): Bird {
  // ...
}

你认为你可以将这些传递给 clone 吗？

typescript
clone(animalToBird) // OK
clone(crowToBird)   // 错误 TS2345: 类型 '(c: Crow) => Bird' 的参数
                    // 不能分配给类型 '(b: Bird) => Bird' 的参数。
                    // 类型 'Bird' 不能分配给类型 'Crow'。

等等，为什么 animalToBird 可以，但 crowToBird 不行？让我们想想这里发生了什么。

clone 期望一个函数，该函数可以处理任何 Bird。如果我们传递 animalToBird，那很好，因为 animalToBird 可以处理任何 Animal，而 Bird 是 Animal 的子类型，所以 animalToBird 当然可以处理任何 Bird。

但是如果我们传递 crowToBird，那就有问题了：crowToBird 只能处理 Crow，但 clone 可能会传递任何类型的 Bird 给它！如果 clone 传递一个普通的 Bird（不是 Crow）给 crowToBird，那么 crowToBird 可能会尝试调用 .caw()，但普通的 Bird 没有 .caw() 方法，所以我们会得到运行时错误。

这就是为什么函数参数类型是逆变的：为了让函数 A 可分配给函数 B，A 的每个参数必须 >: B 的相应参数。

总结一下，我们有：

  1. A 的 this 类型要么未指定，要么 >: B 的 this 类型
  2. A 的每个参数都 >: B 中相应的参数
  3. A 的返回类型 <: B 的返回类型

这就是为什么函数参数类型是逆变的：为了让函数 A 可分配给函数 B，A 的每个参数必须 >: B 的相应参数。

规则1与简单类型完全相同（如果 A 是枚举 B 的成员，那么 A 的类型是 B，所以我们说的是 B <: B）。

规则2是处理枚举的便利。正如我们在"枚举"中谈到的，规则2是 TypeScript 中不安全的一大来源，这是我建议抛弃 baby 和避免使用 bathwater 并完全避免枚举的一个原因。

类型拓宽 ​

类型拓宽是理解 TypeScript 类型推断工作原理的关键。一般来说，TypeScript 会在你声明变量时尽可能推断出最具体的类型，但会选择更通用的类型而不是最具体的类型。这使你的生活作为程序员更容易，并意味着更少的类型检查器投诉。

在第3章中，你已经看到了一些类型拓宽的实例。让我们看一些更多的例子。

当你声明一个变量时，允许它稍后被修改（例如，使用 let 或 var），其类型从其字面值拓宽到该字面值所属的基本类型：

typescript
let a = 'x'        // string
let b = 3          // number
let c = true       // boolean
const d = {x: 3}   // {x: number}

enum E {X, Y, Z}
let e = E.X        // E

但对于不可变声明：

typescript
const a = 'x'      // 'x'
const b = 3        // 3
const c = true     // true

enum E {X, Y, Z}
const e = E.X      // E.X

你可以使用显式类型注解来防止你的类型被拓宽：

typescript
let a: 'x' = 'x'   // 'x'
let b: 3 = 3       // 3
var c: true = true // true
const d: {x: 3} = {x: 3} // {x: 3}

当你重新分配一个非拓宽类型使用 let 或 var 时，TypeScript 会为你拓宽它。要告诉 TypeScript 保持它窄，添加一个显式类型注解到你的原始声明：

typescript
const a = 'x'      // 'x'
let b = a          // string

const c: 'x' = 'x' // 'x'
let d = c          // 'x'

初始化为 null 或 undefined 的变量被拓宽为 any：

typescript
let a = null       // any
a = 3              // any
a = 'b'            // any

但当一个初始化为 null 或 undefined 的变量离开它被声明的作用域时，TypeScript 分配一个确定的类型：

typescript
function x() {
  let a = null     // any
  a = 3            // any
  a = 'b'          // any
  return a
}

x()                // string

const 类型 ​

TypeScript 带有一个特殊的 const 类型，你可以用来选择退出类型拓宽声明在一个时间（阅读"类型断言"）：

typescript
let a = {x: 3}     // {x: number}
let b: {x: 3} = {x: 3} // {x: 3}
let c = {x: 3} as const // {readonly x: 3}

const 选择退出类型拓宽并递归地将其成员标记为 readonly，即使对于深度嵌套的数据结构：

typescript
let d = [1, {x: 2}]        // (number | {x: number})[]
let e = [1, {x: 2}] as const // readonly [1, {readonly x: 2}]

使用 as const 当你希望 TypeScript 推断你的类型尽可能窄。

过度属性检查 ​

类型拓宽也会在 TypeScript 检查一个对象是否可以分配给另一个对象类型时发挥作用。

回想一下"形状和数组变异性"中的内容，对象类型在其成员中是协变的。但如果 TypeScript 坚持这种额外检查，它可能会变得有点过于严格。

例如，考虑一个你可能传递给类来配置它的 Options 对象：

typescript
type Options = {
  baseURL: string
  cacheSize?: number
  tier?: 'prod' | 'dev'
}

class API {
  constructor(private options: Options) {}
}

现在，如果你拼错了一个选项会发生什么？

typescript
new API({
  baseURL: 'https://api.mysite.com',
  tier: 'prod'
  // 错误 TS2345: 类型 '{tier: string}' 的参数
  // 不能分配给类型 'Options' 的参数。
  // 对象字面量只能指定已知属性，
  // 但 'tier' 在类型 'Options' 中不存在。
  // 你是想写 'tier' 吗？
})

这是在使用 JavaScript 时的常见错误，所以 TypeScript 帮助我们捕获它真的很有用。但如果对象类型在其成员中是协变的，TypeScript 如何捕获这个？

那就是：

• 我们期望类型 {baseURL: string, cacheSize?: number, tier?: 'prod' | 'dev'}。

• 我们传入类型 {baseURL: string, tier: string}。

• 我们传入的类型是我们期望的子类型，但不知何故，TypeScript 知道报告错误。

TypeScript 能够捕获这个是由于其过度属性检查，它的工作原理如下：当你尝试将新鲜对象字面量类型 T 分配给另一个类型 U 时，如果 T 有 U 中不存在的属性，那么 TypeScript 报告错误。

新鲜对象字面量类型是 TypeScript 从对象字面量推断的类型。如果该对象字面量要么被分配给变量，要么我们不做类型断言（见"类型断言"），那么新鲜对象字面量类型被拓宽为常规对象类型，其新鲜度消失。

这个定义很密集，所以让我们通过我们的例子再次走一遍，尝试一些更多的变化主题：

typescript
type Options = {
  baseURL: string
  cacheSize?: number
  tier?: 'prod' | 'dev'
}

class API {
  constructor(private options: Options) {}
}

new API({
  baseURL: 'https://api.mysite.com',
  tier: 'prod'
})

new API({ // 错误 TS2345: 类型 '{baseURL: string; badTier: string}' 的参数
  baseURL: 'https://api.mysite.com',  // 不能分配给类型 'Options' 的参数
  badTier: 'prod'
})

let badOptions = {
  baseURL: 'https://api.mysite.com',
  badTier: 'prod'
}

new API(badOptions) // OK

let options: Options = { // 错误 TS2322: 类型 '{baseURL: string; badTier: string}'
  baseURL: 'https://api.mysite.com',  // 不能分配给类型 'Options'
  badTier: 'prod'
}

new API(options)

• 我们用 baseURL 和我们两个可选属性之一 tier 实例化 API。这按预期工作。

• 这里，我们将 tier 拼错为 badTier。我们传递给 new API 的选项对象是新鲜的（因为类型被推断为对象字面量，我们没有做类型断言），所以 TypeScript 对其运行过度属性检查，揭示过度的 badTier 属性（在我们的选项对象中定义但不在 Options 类型中）。

• 我们将选项对象分配给变量 badOptions。TypeScript 不再考虑它是新鲜的，并退出过度属性检查；没有错误。

• 当我们显式地将 options 类型化为 Options 时，我们分配给 options 的对象是新鲜的，所以 TypeScript 对其执行过度属性检查。注意这种情况下过度属性检查不会在我们将 options 传递给 new API 时发生；相反，它发生在我们将 options 对象分配给变量 options 时。

不要担心——你不需要记住这些规则。它们是 TypeScript 的内部启发式方法，用于捕获程序员在实际工作中最常见的错误，所以不要成为你的负担，作为程序员。只要记住它们在你想知道 TypeScript 如何抱怨你的一个错误时，即使你的公司代码库和主代码审查员都没有注意到。

细化 ​

TypeScript 执行基于流的类型推断^2，这是一种符号执行^1，其中类型检查器使用控制流语句如 if、?:、||、switch 以及类型查询，通过代码跟踪类型。这是一个非常方便的功能，对于类型检查器来说，但它是另一个少数语言支持的功能之一。

让我们通过一个例子来演示。假设我们在 TypeScript 中构建了一个 API 来使用 CSS 规则，并且一个同事想要使用它来设置 HTML 元素的宽度。他们传入宽度，我们然后想要解析为值和单位：

我们首先实现一个函数来解析 CSS 字符串为值和单位：

typescript
// 我们使用字符串字面量的联合来描述
// CSS 可能值的一些可能值
type Unit = 'cm' | 'px' | '%'

// 将字符串转换为单位
let units: Unit[] = ['cm', 'px', '%']

// 检查每个单位，如果没有匹配则返回 null
function parseUnit(value: string): Unit | null {
  for (let i = 0; i \< units.length; i++) {
    if (value.endsWith(units[i])) {
      return units[i]
    }
  }
  return null
}

我们然后可以使用 parseUnit 来解析用户传递给我们的宽度值，宽度可能是数字（我们假设以像素为单位），或带有单位的字符串，或者可能是 null 或 undefined。

我们利用类型细化的几次在这个例子中：

typescript
type Width = {
  unit: Unit,
  value: number
}

function parseWidth(width: number | string | null | undefined): Width | null {
  // 如果 width 是 null 或 undefined，提前返回
  if (width == null) { // ①
    return null
  }

  // 如果 width 是数字，假设像素
  if (typeof width === 'number') { // ②
    return {unit: 'px', value: width}
  }

  // 尝试从 width 解析单位
  let unit = parseUnit(width)
  if (unit) { // ③
    return {unit, value: parseFloat(width)}
  }

  // 否则，返回 null
  return null // ④
}

• TypeScript 足够聪明，知道对 null 进行松散相等检查将对 null 和 undefined 在 JavaScript 中都返回 true。它知道如果这个检查通过，那么我们将返回，如果没有通过，那么从那时起 width 的类型是 number | string（它不能是 null 或 undefined 了）。我们说类型被从 number | string | null | undefined 细化为 number | string。

• typeof 检查查询值在运行时看到其类型是什么。TypeScript 利用 typeof 在编译时：在 if 分支中检查通过的地方，TypeScript 知道 width 的类型是 number。在 if 分支之外（因为我们有 return），TypeScript 知道 width 必须是 string（它不能是 number、null 或 undefined 了）。

• 因为调用 parseUnit 可能返回 null，我们检查它是否为真。TypeScript 知道如果 unit 为真，那么它必须是 Unit 类型在 if 分支中——否则，unit 必须是假的^3，意味着它必须是类型 null（定义为 Unit | null）。

• 最后，我们返回 null。这只能发生如果用户传递了 width 的字符串，但那个字符串包含了我们不支持的单位。

我已经详细说明了 TypeScript 对每个类型细化执行的确切思考，但我希望这能让你了解程序员在阅读代码时的想法。TypeScript 在获取你的思维并将其结晶为类型检查和推断规则方面做得非常出色。

判别联合类型 ​

正如我们刚刚学到的，TypeScript 对 JavaScript 的工作原理有深刻的理解，并且能够跟随你的程序，就像你世界中最聪明的程序员在阅读你的代码一样。让我们看看当我们通过你的程序跟踪类型时，TypeScript 如何表现。

例如，假设我们正在为应用程序构建自定义事件系统。我们首先定义一对事件类型，用户可以触发的事件（比如，用户在文本输入中键入了某些内容的 UserTextEvent 和用户将鼠标移动到坐标 [100, 200] 的 UserMouseEvent）：

typescript
type UserTextEvent = {value: string}
type UserMouseEvent = {value: [number, number]}

type UserEvent = UserTextEvent | UserMouseEvent

function handle(event: UserEvent) {
  if (typeof event.value === 'string') {
    event.value    // string
    // ...
    return
  }
  event.value      // [number, number]
}

在 if 块内，TypeScript 知道 event.value 必须是 string（因为 typeof 检查），这意味着在 if 块之后 event.value 是 [number, number]（因为 return 在 if 块中）。

当发生这种情况时，我们的事件类型变得复杂化了。让我们添加一些更多信息到我们的事件类型，看看 TypeScript 如何处理我们细化我们的类型：

typescript
type UserTextEvent = {type: 'TextEvent', value: string, target: HTMLInputElement}
type UserMouseEvent = {type: 'MouseEvent', value: [number, number],
                      target: HTMLElement}

type UserEvent = UserTextEvent | UserMouseEvent

function handle(event: UserEvent) {
  if (event.type === 'TextEvent') {
    event.value    // string
    event.target   // HTMLInputElement
    // ...
    return
  }
  event.value      // [number, number]
  event.target     // HTMLElement
}

现在当我们基于其标记字段（event.type）的值细化事件时，TypeScript 知道在 if 分支中事件必须是 UserTextEvent，在 if 分支之后它必须是 UserMouseEvent。

这种模式被称为判别联合类型。使用判别联合类型的好方法是：

• 在联合类型的同一个地方的每个情况下。这意味着同一个对象字段，或者同一个数组索引，或者同一个元组类型。在实践中，标记联合通常使用对象类型。

• 类型为字面量类型（字面量字符串、数字、布尔值等）。你不能用联合匹配不同类型的字面量类型；通常，最好坚持单一类型；通常，那是字符串字面量类型。

• 不接受任何泛型类型参数。

• 互斥的（即，在联合类型内唯一）。

有了这些想法，让我们更新我们的事件类型。

完整性（Totality） ​

程序员在她的床边桌子上放了两个杯子，然后去睡觉：一个满的，以防她渴了，一个空的，以防她不渴。

由于标签是联合类型的每个成员唯一的，TypeScript 知道这两个是互斥的。

完整性，也称为穷尽性检查（exhaustiveness checking），是类型检查器确保你覆盖了联合类型的所有情况，并在你遗漏某些情况时给出有用警告的能力。它来自 Haskell、OCaml 和其他语言，基于模式匹配。

TypeScript 对完整性检查有很好的支持。它会检查你是否覆盖了联合类型的所有情况，并在你遗漏某些情况时给出有用的警告。例如，它们在处理 Flux 动作、Redux 减速器或 React 的 useReducer 时非常有价值。

让我们使用标记联合来编写一个处理不同情况的函数，并看看 TypeScript 如何帮助我们确保覆盖所有情况：

typescript
type Weekday = 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'
type Day = Weekday | 'Sat' | 'Sun'

function getNextDay(w: Weekday): Day {
  switch (w) {
    case 'Mon': return 'Tue'
  }
}

我们明显遗漏了一些情况！TypeScript 会给我们一个错误：

Error TS2366: Function lacks ending return statement and
return type does not include 'undefined'.

TypeScript 检查了我们的 switch 语句，发现我们没有覆盖 Weekday 的所有情况，并且由于我们注释了 getNextDay 返回 Day（而不是 Day | undefined），TypeScript 警告我们。

实现细节在这个例子中并不重要；无论你使用什么样的控制结构——switch、if、三元运算符等等——TypeScript 都会监视你的后背，确保你覆盖了每种情况。

这是一个非常有用的功能，用于防止真正的错误。例如：

typescript
type Weekday = 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'

function isBigDay(day: Weekday): boolean {
  switch (day) {
    case 'Mon': return false
    case 'Tue': return true
    case 'Wed': return false
    case 'Thu': return false
    case 'Fri': return true
  }
}

也许客户端的某个地方有你忘记的周末营业时间，你意识到你需要将周末添加到你的 Day 类型中：

typescript
let nextDay = {
  Mon: 'Tue',
  Tue: 'Wed',
  Wed: 'Thu',
  Thu: 'Fri',
  Fri: 'Sat'
} as const

回想一下"Totality"中的 weekDay 示例，有两种方式来强制对象定义一组特定的键。Record 类型是第一种。

也许你忘记了更新你的 isBigDay 函数，但 TypeScript 不会忘记——TypeScript 会给你一个错误，提醒你更新你的函数以处理新的情况。

TSC 标志：noImplicitReturns ​

为了让 TypeScript 检查你的函数是否在所有代码路径上都返回值，你需要在你的 tsconfig.json 中启用 noImplicitReturns 标志：

json
{
  "compilerOptions": {
    "noImplicitReturns": true
  }
}

或者，你可以在命令行上使用这个标志：

bash
tsc --noImplicitReturns

这个标志确保你的函数在所有代码路径上都有返回语句，这对于完整性检查特别有用。

高级完整性示例 ​

让我们看一个更复杂的例子，展示 TypeScript 如何帮助我们处理更复杂的联合类型：

typescript
type Weekday = 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'

// 使用对象映射的方式
let nextDay = {
  Mon: 'Tue',
  Tue: 'Wed',
  Wed: 'Thu',
  Thu: 'Fri',
  Fri: 'Sat'
} as const

nextDay.Mon // 'Tue'

如果你后来决定添加周末到你的类型中，TypeScript 会提醒你更新所有相关的代码。这种完整性检查是防止运行时错误的强大工具。

也许你忘记了更新你的 nextDay 对象来处理新的情况，但 TypeScript 不会忘记——TypeScript 会给你一个错误，提醒你更新你的对象以处理新的情况。

这种完整性检查在处理状态机、事件处理器和其他需要处理所有可能情况的场景中特别有价值。

对象类型的类型操作符 ​

对象是 JavaScript 的核心，TypeScript 为你提供了一整套表达和安全操作它们的方法。

记住联合（|）和交集（&）这两个类型操作符，我在"联合和交集类型"中介绍过？事实证明，它们不是 TypeScript 给你的唯一类型操作符！让我们来看看一些专门用于处理形状的更多类型操作符。

keying-in 操作符 ​

假设你有一个复杂的嵌套类型来建模你从社交媒体 API 获得的 GraphQL API 响应：

typescript
type APIResponse = {
  user: {
    userId: string
    friendList: {
      count: number
      friends: {
        firstName: string
        lastName: string
      }[]
    }
  }
}

你可能从 API 获取该响应，然后渲染它：

typescript
function getAPIResponse(): Promise\<APIResponse> {
  // ...
}

function renderFriendList(friendList: unknown) {
  // ...
}

let response = await getAPIResponse()
renderFriendList(response.user.friendList)

friendList 的类型应该是什么？（它的子对象现在被标记为 unknown。）你可以输入它并重新实现你的顶级 APIResponse 类型：

typescript
type APIResponse = {
  user: {
    userId: string
    friendList: {
      count: number
      friends: {
        firstName: string
        lastName: string
      }[]
    }
  }
}

function renderFriendList(friendList: FriendList) {
  // ...
}

但是你必须为你的每个顶级类型想出名称，这你并不总是想要（例如，如果你想从你的 GraphQL 模式自动生成 TypeScript 类型）。相反，你可以 key in 到你的类型：

typescript
type APIResponse = {
  user: {
    userId: string
    friendList: {
      count: number
      friends: {
        firstName: string
        lastName: string
      }[]
    }
  }
}

type FriendList = APIResponse['user']['friendList']

function renderFriendList(friendList: FriendList) {
  // ...
}

你可以 key in 到任何形状（对象、类构造函数或类实例）和任何数组。例如，要获取单个朋友的类型：

typescript
type Friend = FriendList['friends'][number]

number 是一种 key in 到数组类型的方法；对于元组，使用 0、1 或另一个数字字面量类型来表示你想要 key in 到的索引。

keying-in 的语法故意类似于你在常规 JavaScript 对象中查找值的方式——只是你查找的是类型而不是值，所以你可以在形状中查找类型。

注意你必须使用方括号表示法，而不是点表示法，来查找属性类型，当 keying in 时。

keyof 操作符 ​

使用 keyof 获取对象所有键的联合字符串字面量类型。使用前面的 APIResponse 示例：

typescript
type ResponseKeys = keyof APIResponse // 'user'
type UserKeys = keyof APIResponse['user'] // 'userId' | 'friendList'
type FriendListKeys =
  keyof APIResponse['user']['friendList'] // 'count' | 'friends'

结合 keying-in 和 keyof 操作符，你可以实现一个类型安全的 getter 函数，该函数在给定键的对象中查找值：

typescript
function get\<
  O extends object,
  K extends keyof O
>(
  o: O,
  k: K
): O[K] {
  return o[k]
}

• get 是一个接受对象和键 k 的函数。

• keyof O 是字符串字面量类型的联合，扩展了 O 的所有键。泛型类型 K 扩展——并且是该联合的子类型。例如，如果 O 有类型 {a: number, b: string, c: boolean}，那么 keyof O 将是类型 'a' | 'b' | 'c'，任何扩展 keyof O 的子类型都可以是类型 'a'、'b'、'c' 或任何其他 keyof O 的子类型。

• O[K] 是你在查找 K 时得到的类型。继续我们的示例，如果 K 是 'a'，那么我们知道在编译时 get 返回一个数字。或者，如果 K 是 'b' | 'c'，那么我们知道 get 返回 string | boolean。

这些类型操作符的酷炫之处在于它们如何精确和安全地让你描述形状类型：

typescript
type ActivityLog = {
  lastEvent: Date
  events: {
    id: string
    timestamp: Date
    type: 'Read' | 'Write'
  }[]
}

let activityLog: ActivityLog = // ...
let lastEvent = get(activityLog, 'lastEvent') // Date

TypeScript 为你工作，在编译时验证 lastEvent 的类型是 Date。当然，你可以扩展这个以便 key in 到对象更深层次。让我们重载 get 以接受最多三个键：

typescript
type Get = {
  \<
    O extends object,
    K1 extends keyof O
  >(o: O, k1: K1): O[K1]
  \<
    O extends object,
    K1 extends keyof O,
    K2 extends keyof O[K1]
  >(o: O, k1: K1, k2: K2): O[K1][K2]
  \<
    O extends object,
    K1 extends keyof O,
    K2 extends keyof O[K1],
    K3 extends keyof O[K1][K2]
  >(o: O, k1: K1, k2: K2, k3: K3): O[K1][K2][K3]
}

let get: Get = (object: any, ...keys: string[]) => {
  let result = object
  keys.forEach(k => result = result[k])
  return result
}

get(activityLog, 'events', 0, 'type') // 'Read' | 'Write'

我们声明一个重载函数签名，get 有三种情况，当我们调用 get 时：

• 用一个键，两个键和三个键。

• 这个一键情况就像前面的例子：O 是 object 的子类型，K1 是 O 的键的子类型，返回类型是你在用 K1 keying 到 O 时得到的任何特定类型。

• 两键情况类似于一键情况，但我们声明一个更多的泛型类型，K2，来建模可能的键在嵌套对象中，该对象从用 K1 keying 到 O 得到。

• 我们通过 keying 两次构建——我们首先得到 O[K1] 的类型，然后得到 [K2] 的类型。

• 对于这个例子，我们处理最多三个嵌套键；如果你正在编写一个真实世界的库，你可能想要处理更多的情况。

TSC 标志：KEYOFSTRINGSONLY

在 JavaScript 中，对象不仅可以有字符串键，还可以有数字键，这些键在运行时被强制转换为字符串。

默认情况下，当 TypeScript 返回 keyof 类型的值时，它返回 number | string | symbol，通过约定，我们通常使用数字键作为数组，这些键在运行时被强制转换为字符串。

这种行为是正确的，但可能使使用 keyof 变得棘手，因为你可能必须证明给 TypeScript 你正在操作的特定键是 string，而不是 number 或 symbol。

要选择 TypeScript 的遗留行为——其中键必须是字符串——启用 keyofStringsOnly tsconfig.json 标志。

Record 类型 ​

TypeScript 的内置 Record 类型是一种描述对象作为从某物到某物的映射的方式。

回想一下"Totality"中的 weekDay 示例，有两种方式来强制对象定义一组特定的键。Record 类型是第一种。

让我们使用 Record 来构建一个从一周中每一天到下一天的映射。使用 Record，你可以对 nextDay 中的键和值施加一些约束：

typescript
type Weekday = 'Mon' | 'Tue' | 'Wed' | 'Thu' | 'Fri'
type Day = Weekday | 'Sat' | 'Sun'

let nextDay: Record\<Weekday, Day> = {
  Mon: 'Tue'
}

现在，你得到一个很好的、有用的错误消息：

Error TS2739: Type '{Mon: "Tue"}' is missing the following properties
from type 'Record\<Weekday, Day>': Tue, Wed, Thu, Fri.

向你的对象添加缺失的 weekdays，当然，会使错误消失。

Record 给你比常规对象索引签名更多的自由度：使用常规索引签名，你可以约束对象值的类型，但键只能是常规的 string、number 或 symbol；使用 Record，你还可以约束对象键到 string 和 number 的子类型。

Mapped Types ​

TypeScript 给你第二种、更强大的方式来声明一个更安全的 nextDay 类型：mapped types。让我们使用 mapped types 来说 nextDay 是一个对象，其中每个 weekday 都有一个键，其值是 Day：

typescript
let nextDay: {[K in Weekday]: Day} = {
  Mon: 'Tue'
}

这是另一种获得有用提示的方法，说明你遗漏了什么：

Error TS2739: Type '{Mon: "Tue"}' is missing the following properties
from type '{Mon: Weekday; Tue: Weekday; Wed: Weekday; Thu: Weekday;
Fri: Weekday}': Tue, Wed, Thu, Fri.

Mapped types 是 TypeScript 独有的语言特性。就像字面类型一样，它们是一个实用特性，只是让你的生活更轻松。

正如你所看到的，mapped types 有自己的特殊语法。就像索引签名一样，你可以对映射类型中的值类型施加约束，但键名称。事实上，TypeScript 使用 mapped types 来实现其内置的 Record 类型，因为它更强大：

typescript
type Record\<K extends keyof any, T> = {
  [P in K]: T
}

Mapped types 给你比仅仅 Record 更多的力量，因为除了让你给对象的键和值施加类型外，当你将它们与 keyed-in types 结合时，它们让你对哪个值类型对应哪个键名称施加约束。

让我们快速浏览一些你可以用 mapped types 做的事情。

我们通过映射 Account 创建一个新的对象类型 OptionalAccount，将每个字段标记为可选：

typescript
type OptionalAccount = {
  [K in keyof Account]?: Account[K]
}

我们通过映射 Account 创建一个新的对象类型 NullableAccount，添加 null 作为每个字段的可能值：

typescript
type NullableAccount = {
  [K in keyof Account]: Account[K] | null
}

我们通过映射 Account 创建一个新的对象类型 ReadonlyAccount，使每个字段只读（即可读但不可写）：

typescript
type ReadonlyAccount = {
  readonly [K in keyof Account]: Account[K]
}

我们通过映射 Account 创建一个新的对象类型 Account2，等价于我们的原始 Account 类型，通过映射 OptionalAccount 并移除 readonly 修饰符：

typescript
type Account2 = {
  [K in keyof OptionalAccount]-?: OptionalAccount[K]
}

我们通过映射 OptionalAccount 创建一个新的对象类型 Account3，等价于我们的原始 Account 类型，通过映射 OptionalAccount 并移除可选 (?) 操作符：

typescript
type Account3 = {
  -readonly [K in keyof ReadonlyAccount]: ReadonlyAccount[K]
}

你可以标记字段为可选 (?) 或必需 (-?)，以及可写或只读 (readonly)。你也可以取消标记它们。使用 minus (-) 操作符——一个特殊的操作符，只在映射类型中可用——我们可以取消标记字段为必需和可写，分别。

注意

Minus (-) 是一个对应的 plus (+) 操作符。你可能永远不会使用这个操作符直接，因为它隐含在映射类型中，但它只是为了完整性。+readonly 等价于 readonly，+? 等价于 ?。

伴随对象模式 ​

伴随对象模式来自 Scala，是一种将类型和对象配对的方式，这些对象和类共享相同的名称。在 TypeScript 中，有一个类似的模式，同样有用——我们也称之为伴随对象模式——我们可以用它来配对一个类型和一个对象。

它看起来像这样：

typescript
type Currency = {
  unit: 'EUR' | 'GBP' | 'JPY' | 'USD'
  value: number
}

let Currency = {
  DEFAULT: 'USD',
  fromValue(value: number, unit = Currency.DEFAULT): Currency {
    return {unit, value}
  }
}

这个模式在 TypeScript 中非常有用，因为它让你将类型和值信息分组在一个名称下。在这个例子中，消费者可以导入 Currency，并使用它作为类型和值：

typescript
import {Currency} from './Currency'

let amountDue: Currency = {  // Currency 作为类型
  unit: 'JPY',
  value: 83733.10
}

let otherAmountDue = Currency.fromValue(330, 'EUR')  // Currency 作为值

因为 TypeScript 有单独的类型和值命名空间，这种模式是安全的，不会导致命名冲突。

高级函数类型 ​

让我们看一些更高级的技术，这些技术经常与函数类型一起使用。

改进元组的类型推断 ​

当你在 TypeScript 中声明一个元组时，TypeScript 会根据你给它的内容推断该元组的类型。它会推断元组的长度以及每个位置包含的类型：

typescript
let a = [1, true] // (number | boolean)[]

但有时你想要推断更严格的，将 a 视为固定长度的元组而不是数组。你可以使用 as const 断言（"类型断言"）来做到这一点，或者你可以使用一个 as 断言来将元组的类型转换为元组类型（更多关于这个在"类型断言"中）：

typescript
let a = [1, true] as const // readonly [1, true]

如果你想要类型化你的元组作为元组，但避免类型断言，并避免将元组的类型缩小到尽可能窄的推断，你可以利用这种技术来利用 TypeScript 的类型推断：

typescript
function tuple<T extends readonly unknown[]>(
  ...ts: T
): T {
  return ts
}

let a = tuple(1, true) // [1, true]

使用这种技术来利用 TypeScript 的类型推断，以避免在你的代码使用大量元组类型时进行类型断言。

用户定义的类型守卫 ​

对于某些类型的布尔返回函数，简单地说你的函数返回 boolean 可能不够。例如，让我们编写一个函数，告诉你是否传递了一个 string 或 not：

typescript
function isString(a: unknown): boolean {
  return typeof a === 'string'
}

isString('a') // 评估为 true isString([7]) // 评估为 false

到目前为止一切都很好。如果你尝试在一些真实世界的代码中使用 isString，会发生什么？

typescript
function parseInput(input: string | number) {
  let formattedInput: string
  if (isString(input)) {
    formattedInput = input.toUpperCase() // 错误 TS2339: 属性 'toUpperCase' 在类型 'string | number' 上不存在
  }
}

所以这不太好。发生了什么？你尝试在一些真实世界的代码中使用 isString，但 TypeScript 不知道调用 isString(input) 实际上告诉我们关于 input 的类型的任何信息。从 TypeScript 的角度来看，isString 只是一个返回 boolean 的函数；TypeScript 不知道当 isString 返回 true 时，这意味着传递给 isString 的参数是 string。

为了表达这一点，我们使用一种叫做用户定义类型守卫的东西：

typescript
function isString(a: unknown): a is string {
  return typeof a === 'string'
}

注意我们如何将返回类型从 boolean 改为 a is string。a is string 是一个类型谓词，它是一种特殊的类型，告诉 TypeScript 类型细化器，如果 isString 返回 true，那么传递给 isString 的任何参数都是 string 类型。

现在，当我们在真实世界的代码中使用 isString 时，TypeScript 知道在 if 分支中细化 input 的类型：

typescript
function parseInput(input: string | number) {
  let formattedInput: string
  if (isString(input)) {
    formattedInput = input.toUpperCase() // OK，input 是 string
  }
}

用户定义的类型守卫是一个强大的功能，让你恢复一些你在使用高度动态代码时可能失去的类型安全性。它们在处理来自第三方 API 的数据、用户输入或任何你不能 100% 确定类型的情况时特别有用。

什么给了？如果 typeof 对常规类型细化有效（参见"细化"），为什么它在这里不起作用？

关于类型细化的事情是它只足够强大，可以细化你所在范围内的变量类型。一旦你离开该范围，细化就不会延续到你所在的任何新范围。在我们的实现中，我们细化了输入参数的类型为 string，但因为类型细化不会延续到新范围，它就丢失了——所有 TypeScript 知道的是 isString 返回了一个 boolean。

我们能做的是告诉类型检查器，不仅 isString 返回一个 boolean，而且无论何时 boolean 是 true，我们传递给 isString 的参数都是 string。为了做到这一点，我们使用一种叫做用户定义类型守卫的东西：

typescript
function isString(a: unknown): a is string {
  return typeof a === 'string'
}

类型守卫是 TypeScript 中的内置功能，是让你使用 typeof 和 instanceof 细化类型的原因。但有时，你需要声明类型守卫来确保细化在你使用该函数时得以延续。

用户定义的类型守卫不限于单个参数，但它们也不限于简单类型：

typescript
type LegacyDialog = // ...
type Dialog = // ...

function isLegacyDialog(
  dialog: LegacyDialog | Dialog
): dialog is LegacyDialog {
  // ...
}

你不会经常使用用户定义的类型守卫，但当你使用时，它们对于编写干净、可重用的代码非常棒。没有它们，你必须内联所有的 typeof 和 instanceof 类型守卫，而不是将 isString 提取到执行相同检查的更好封装、更可重用的方式。

条件类型 ​

条件类型可能是 TypeScript 中所有功能中最独特的功能。在高层次上，条件类型让你说，"声明一个依赖于类型 U 和 V 的类型 T，如果 <: V，那么将 T 分配给 A，否则，将 T 分配给 B。"

在代码中它可能看起来像这样：

typescript
type IsString<T> = T extends string ?
  true :
  false

type A = IsString<string> // true
type B = IsString<number> // false

让我们逐行分解：

• 我们声明一个新的条件类型 IsString，它接受一个泛型类型 T。"条件"部分是这个条件类型是否是 string 的子类型；也就是说，"T 是 string 的子类型吗？"

• 如果 T 是 string 的子类型，我们解析为类型 true。

• 否则，我们解析为类型 false。

注意这里的语法如何类似于常规值级三元表达式，但在类型级别。就像常规三元表达式一样，你也可以嵌套它们。

条件类型几乎可以在任何地方使用类型——在类型别名、接口、类、参数类型和泛型默认值中的函数和方法。

分布式条件类型 ​

虽然你可以在 TypeScript 中表达简单的条件，如我们刚才看到的例子，但在各种方式中——使用条件类型、重载函数签名和映射类型——条件类型让你做更多的事情。原因是它们遵循分布式法则（记住，来自代数类？）。这意味着如果你有一个条件类型，那么表达式在右边等价于左边的那些：

(string | number) extends T ? A : B(string extends T ? A : B) | (number extends T ? A : B)

我知道，我知道，你不应该跳过这本书来学习数学——你来这里是为了类型。所以让我们更具体一些。假设我们有一个函数，它接受某种类型 T 的变量，并将其提升到类型 T 的数组。如果我们传入一个联合类型给 T 会发生什么？

typescript
type ToArray<T> = T[]
type A = ToArray<number> // number[]
type B = ToArray<number | string> // (number | string)[]

很直接。现在如果我们添加一个条件类型会发生什么？（注意条件实际上没有做任何事情，因为两个分支都解析为相同的类型 T[]；它只是在这里告诉 TypeScript 在元组类型上分布。）看一下：

typescript
type ToArray<T> = T extends unknown ? T[] : T[]
type A = ToArray<number> // number[]
type B = ToArray<number | string> // number[] | string[]

你抓住了吗？当你使用条件类型时，TypeScript 会在联合类型上分布条件的分支。让我们看看 TypeScript 如何计算这个类型：

  1. 从输入开始：

    typescript
    type A = ToArray<number | string, boolean>

  2. 在联合上分布条件：

    typescript
    type A = ToArray<number, boolean> | ToArray<string, boolean>

  3. 在 ToArray 的定义中替换并应用 T 和 U：

    typescript
    type A = (number extends unknown ? number[] : never) |
         (string extends unknown ? string[] : never)

  4. 评估条件：

    typescript
    type A = number[] | string[]

  5. 简化：

    typescript
    type A = number[] | string[]

为什么这很重要？嗯，它让你安全地表达一堆常见操作。

typescript
type Without<T, U> = T extends U ? never : T

你可以像这样使用 Without：

typescript

  boolean

    typescript
    type A = Without<boolean | number | string, boolean>

  2. 在联合上分布条件：

    typescript
    type A = Without<boolean, boolean> |
         Without<number, boolean> |
         Without<string, boolean>

  3. 在 Without 的定义中替换并应用 T 和 U：

typescript
type ElementType<T> = T extends unknown[] ? T[number] : T
type A = ElementType<number[]> // number

现在，让我们使用 infer 重写它：

typescript
type ElementType2<T> = T extends (infer U)[] ? U : T
type B = ElementType2<number[]> // number

在这个简单例子中，ElementType 等价于 ElementType2。注意 infer 子句如何声明一个新的类型变量 U——TypeScript 会从上下文中推断 U 的类型，基于你传递给 ElementType2 的内容。

类型断言 ​

有时，两种类型可能不够相关，所以你无法断言一种是另一种。为了绕过这个，你可以先断言到 any，然后断言到你想要的类型：

typescript
function formatInput(): string | number {
  // ...
}

let input = formatInput()

// 断言 input 是一个字符串
formatInput(input as string) ①

// 这等价于
formatInput(<string>input) ②

① 我们使用类型断言（as）告诉 TypeScript input 是一个字符串，而不是字符串 | 数字，因为类型会让我们相信。你可能会这样做，例如，如果你知道你的 formatInput 函数总是返回一个字符串。

② 类型断言的传统语法使用尖括号。这两种语法在功能上是等价的。

注意

优先使用 as 语法进行类型断言而不是尖括号（<>）语法。前者是明确的，但后者可能与 TSX 语法冲突（参见"TSX 是 JSX + TypeScript"）。使用 TSLint 的 no-angle-bracket-type-assertion 规则来自动强制执行这个约定。

有时，两种类型可能不够相关，所以你无法断言一种是另一种。为了绕过这个，你可以先断言到 any，然后断言到你想要的类型：

typescript
function addToList(list: string[], item: string): string[] {
  // ...
}

addToList(
  this is really, as any, 'really unsafe' ①
)

① 显然，类型断言是不安全的，你应该尽可能避免使用它们。

非空断言 ​

对于可空类型的特殊情况——即类型 T | null 或 T | null | undefined——TypeScript 有特殊的语法来断言值的类型是 T，而不是 null 或 undefined：

typescript
type Dialog = {
  id?: string
}

function closeDialog(dialog: Dialog) {
  if (!dialog.id) { ①
    return
  }
  setTimeout(() => ②
    removeFromDOM(
      dialog,
      document.getElementById(dialog.id) // Error TS2345: Argument of type
                                        // 'string | undefined' is not assignable
                                        // to parameter of type 'string'. ③
    )
  })
}

function removeFromDOM(dialog: Dialog, element: Element) {
  element.parentNode.removeChild(element) // Error TS2531: Object is possibly
                                         // 'null'. ④
  delete dialog.id
}

① 如果对话框已经被删除（没有 id），我们提前返回。

② 我们从 DOM 中移除对话框，在事件循环的下一轮，这样任何其他依赖于 dialog 的代码都有机会完成运行。

③ 因为我们在函数内部，在 setTimeout 的回调中，TypeScript 不知道某些代码是否在 ① 和 ③ 之间改变了 dialog，所以它使我们在 ① 中做的细化无效。在这种情况下，虽然我们知道如果 dialog 没有 id，我们会提前返回，但 TypeScript 不知道——它只知道如果 dialog 有一个 id，它可能是一个不可为空的 HTMLElement，但 TypeScript 不知道这一点——它只知道如果有一个具有该 id 的元素，document.getElementById 返回一个 HTMLElement，否则返回 null。

④ 类似地，虽然我们知道对话框肯定在 DOM 中并且肯定有一个父 DOM 节点，但 TypeScript 不知道这一点。

解决这个问题的一种方法是添加一堆 if（=== null）检查。虽然这是正确的做法，如果你确定某些东西不是 null，TypeScript 提供了特殊语法，当你确定某些东西不是 null | undefined 时：

typescript
type Dialog = {
  id?: string
}

function closeDialog(dialog: Dialog) {
  if (!dialog.id) {
    return
  }
  setTimeout(() =>
    removeFromDOM(
      dialog,
      document.getElementById(dialog.id!)
    )
  )
}

function removeFromDOM(dialog: Dialog, element: Element) {
  element.parentNode!.removeChild(element)
  delete dialog.id
}

注意非空断言操作符（!）的使用，它告诉 TypeScript 我们确定 dialog.id 的结果是我们的 document.getElementById 调用，element.parentNode 被定义。当一个非空断言被使用时，我们检查 dialog 有一个 id 属性定义——暗示它是一个 VisibleDialog——即使在箭头函数内部，TypeScript 知道对 dialog 的引用没有改变：dialog 在箭头函数内部的类型与在最后一个例子中的函数相同，细化被保留而不是被无效化，就像在最后一个例子中一样。

明确赋值断言 ​

TypeScript 有特殊的语法用于非空断言的特殊情况，用于明确赋值断言对象（作为提醒，明确赋值断言检查是 TypeScript 确保变量在使用之前被赋值的方式）。例如：

typescript
let userId: string
fetchUser()

userId.toUpperCase() // Error TS2454: Variable 'userId' is used
                    // before being assigned.

function fetchUser() {
  userId = globalCache.get('userId')
}

显然，TypeScript 无法预见到调用这个错误。我们声明了变量 userId，但忘记给它赋值，在我们尝试将它转换为大写之前。这本来会是一个运行时错误！注意到了！

但是，如果你的代码看起来更像这样呢？

typescript
let userId: string
fetchUser()

userId.toUpperCase() // Error TS2454: Variable 'userId' is used
                    // before being assigned.

function fetchUser() {
  userId = globalCache.get('userId')
}

我们碰巧有世界上最好的缓存，当我们查询这个缓存时，我们得到一个缓存命中 100% 的时间。所以在调用 fetchUser 后，userId 保证被定义。但 TypeScript 无法自动检测到这一点，所以它仍然抛出同样的错误。我们可以使用明确赋值断言来告诉 TypeScript userId 将被明确赋值，通过我们读取它的时间（注意感叹号）：

typescript
let userId!: string
fetchUser()

userId.toUpperCase() // OK

function fetchUser() {
  userId = globalCache.get('userId')
}

与类型断言和非空断言一样，如果你发现自己使用明确赋值断言很多，你可能做错了什么。

模拟名义类型 ​

到这本书的这一点，如果我想要摇醒你一下，在早上大喊"IS TYPESCRIPT'S TYPE SYSTEM STRUCTURAL!"你会大喊回来"OF COURSE IT'S STRUCTURAL! NOW GET OUT OF MY HOUSE OR I'LL CALL THE POLICE!"那会是一个公平的回应，因为我们一直在敲击这个早期类型系统问题。

撇开现实不谈，有时名义类型确实很有用。例如，假设你有一个有 10 种类型的应用程序，代表处理不同类型对象的不同方式：

typescript
type CompanyID = string
type OrderID = string
type UserID = string
type ID = CompanyID | OrderID | UserID

UserID 类型的值可能是一个简单的哈希，看起来像 "b21bdbf"。所以当你可能别名它为 UserID，在底层它当然只是一个常规的 string。一个接受 UserID 的函数可能看起来像：

typescript
function queryForUser(id: UserID) {
  // ...
}

这是很好的文档，它帮助你团队的其他工程师知道确定哪种类型的 ID 应该传递给 queryForUser。但是 UserID 只是 string 的别名，所以类型系统几乎不会阻止错误。工程师可能意外地传递错误类型的 ID，类型系统不会知道更好：

typescript
let id: CompanyID = "b21bdbf"
queryForUser(id) // OK (!!!)

这就是名义类型派上用场的地方。虽然 TypeScript 不支持开箱即用的名义类型，我们可以用一种叫做类型品牌的技术来模拟它们。^5 类型品牌使你的程序显著更安全。

根据你的应用程序和你的工程团队的大小（团队越大，这种技术越有可能派上用场以防止错误），你可能不需要这样做。

注意

根据你的应用程序和你的工程团队的大小（团队越大，这种技术越有可能派上用场以防止错误），你可能不需要这样做。

首先创建每个名义类型的合成类型品牌：

typescript
type CompanyID = string & {readonly brand: unique symbol}
type OrderID = string & {readonly brand: unique symbol}
type UserID = string & {readonly brand: unique symbol}
type ID = CompanyID | OrderID | UserID

string 和 {readonly brand: unique symbol} 的交集当然是胡言乱语。我选择它是因为不可能自然地构造那种类型，我们唯一创建那种类型值的方法是通过交集。我使用 unique symbol 作为"品牌"，因为它是 TypeScript 中最独特的类型之一；我使用交集与那个品牌和 string，这样我们可以断言给定的 string 是给定的品牌类型。

我们现在需要一种方法来创建 CompanyID、OrderID 和 UserID 类型的值。为了做到这一点，我们将使用伴生对象模式（在"伴生对象模式"中介绍）。我们将为每个品牌类型创建一个构造函数，使用类型断言来构造每个胡言乱语类型的值：

typescript
function CompanyID(id: string) {
  return id as CompanyID
}
function OrderID(id: string) {
  return id as OrderID
}
function UserID(id: string) {
  return id as UserID
}

最后，让我们看看使用这些类型的感觉如何：

typescript
function queryForUser(id: UserID) {
  // ...
}

let companyId = CompanyID("8a9076cf")
let orderId = OrderID("9994acc1")
let userId = UserID("d21d2dbf")

queryForUser(userId)   // OK
queryForUser(companyId) // Error TS2345: Argument of type 'CompanyID' is not
                       // assignable to parameter of type 'UserID'.

这种方法的好处是它几乎没有运行时开销：每个 ID 构造函数只有一个函数调用，这可能会被你的 JavaScript VM 内联。在运行时，每个 ID 只是一个字符串——没有其他编译时的胡言乱语。

再次，对于大多数应用程序，这种方法是过度的。但对于大型应用程序，当与早期类型系统一起工作时，品牌类型可以是一个杀手级安全功能。

安全地扩展原型 ​

当构建 JavaScript 应用程序时，传统认为扩展内置类型的原型是不安全的。这个拇指规则可以追溯到 jQuery 时代，当时 JavaScript 图像构建库如 MooTools 扩展并覆盖内置原型方法。但是当你不是在构建库时，你只是在构建应用程序，JavaScript 给你一个动态语言，让你可以在运行时修改任何内置对象（如 []、push、'abc'、toUpperCase 或 Object.assign）。因为它是一个动态语言，TypeScript 给你静态类型，所以每个内置对象——Array.prototype、Function.prototype、Object.prototype 等等。

虽然回到扩展这些原型是不安全的日子，如果你的代码被静态类型系统如 TypeScript 覆盖，那么你现在可以安全地这样做！^6

例如，我们将在 zip 文件（比如，zip.ts）中扩展 Array 的原型，然后我们增强全局 Array 命名空间以获得我们的新 zip 方法：

typescript
// Tell TypeScript that we're adding zip to Array. We take advantage of inter-
// face merging ("Declaration Merging") to augment the global Array's interface, adding
// our own zip method to the already globally defined interface.
interface Array<T> {
  zip<U>(list: U[]): [T, U][]
}

由于我们的库没有任何显式的导入或导出——意味着它在脚本模式下，如"模块模式与脚本模式"中所述——我们能够通过声明一个与现有 Array 接口完全相同名称的接口来直接增强全局 Array 接口，让 TypeScript 负责为我们合并这两个。如果我们的文件在模块模式下（例如，如果我们需要为我们的 zip 实现导入某些东西），我们必须将我们的全局扩展包装在 declare global 类型声明中（参见"类型声明"^4）：

typescript
declare global {
  interface Array<T> {
    zip<U>(list: U[]): [T, U][]
  }
}

global 是一个特殊的命名空间，包含所有全局定义的值（你可以在模块模式文件中使用而不导入它们；参见第 10 章），让你增强来自模块作用域的名称。

我们然后通过在 Array 的原型上实现 zip 方法来使用 this 类型，这样 TypeScript 正确地推断出我们正在对其调用 zip 的 Array 的类型：

typescript
// We then implement the zip method on Array's prototype. We use a this type so that
// TypeScript correctly infers the Type of the array we're calling .zip on.
// Because TypeScript infers the mapping function's return type as (T | U)[] (TypeScript
// isn't smart enough to realize that it's in fact always a tuple with T in the zeroth index and U
// in the first), we use our tuple utility (from "Improving Type Inference for Tuples") to cre-
// ate a tuple type when calling to a type assertion.

Array.prototype.zip = function<T, U>(
  this: T[],
  list: U[]
): [T, U][] {
  return this.map((v, k) =>
    tuple(v, list[k])
  )
}

注意当我们声明 interface Array<T> 时，我们增强了全局 Array 命名空间为我们整个 TypeScript 项目——意味着包含我们新 zip 的 Array 从我们的文件中可用，TypeScript 将认为 [].zip 是可用的。但为了增强 Array.prototype，我们必须确保无论什么文件加载 zip.ts 首先，为了安装我们方法到 Array.prototype。我们如何确保任何使用 zip 的文件首先加载它？

简单：我们更新我们的 tsconfig.json 来明确排除 zip.ts 从我们的项目，这样消费者必须明确导入它：

json
{
  "compilerOptions": {
    // ...
  },
  "include": [
    "./src"
  ],
  "exclude": [
    "./src/zip.ts"
  ]
}

现在，为了使用我们的新 zip 方法，消费者必须首先导入 zip.ts：

typescript
import './zip'
let zipped = [1, 2, 3].zip(['a', 'b', 'c']) // [T, U][]

现在我们可以使用 zip，完全安全：

typescript
import './zip'

[1, 2, 3]
  .map(n => n * 2)        // number[]
  .zip(['a', 'b', 'c'])   // [number, string][]

运行这个给我们第一次映射的结果，然后压缩数组：

typescript
[
  [2, 'a'],
  [4, 'b'],
  [6, 'c']
]

总结 ​

在这一章中，我们涵盖了 TypeScript 类型系统最高级的特性：从变异的来龙去脉到基于流的类型推断、细化、类型拓宽、总体性和条件类型。我们探索了一些高级类型级编程技术：类型品牌来模拟名义类型，利用条件类型的分布式属性在类型级别进行映射，以及安全地扩展原型。

如果你不理解或不记得所有内容，那没关系——回到这一章，当你在努力如何更安全地表达某些东西时，将其用作参考。
