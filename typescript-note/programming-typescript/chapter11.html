第11章：与 JavaScript 互操作 ​

章节概述 ​

现实世界并不完美。在实际开发中，我们经常面临混合代码库的挑战：既有TypeScript代码，也有JavaScript代码。本章专门讨论如何在这种环境中工作，包括：

  • 使用类型声明为JavaScript代码添加类型
  • 从JavaScript逐步迁移到TypeScript的策略
  • 管理和使用第三方JavaScript库

无论你是想为现有JavaScript项目引入TypeScript，还是需要在TypeScript项目中使用JavaScript库，本章都将为你提供实用的解决方案。

类型声明（Type Declarations） ​

什么是类型声明？ ​

类型声明是一个扩展名为 .d.ts 的文件。与JSDoc注释一样，它是将TypeScript类型附加到原本无类型的JavaScript代码的一种方式。

类型声明的语法与常规TypeScript类似，但有一些区别：

类型声明的三个重要特点 ​

1. 只能包含类型，不能包含值 ​

类型声明只能包含类型，不能包含值。这意味着：

  • ❌ 不能有函数、类、对象或变量的实现
  • ❌ 不能有参数的默认值

2. 可以声明值的存在 ​

虽然类型声明不能定义值，但它们可以声明某个地方存在一个值。我们使用特殊的 declare 关键字来实现这一点。

3. 只声明对外可见的类型 ​

类型声明只为对使用者可见的内容声明类型。我们不为未导出的类型或函数体内的局部变量包含类型。

实际示例：RxJS库的类型声明 ​

让我们看一个来自流行的RxJS库的TypeScript代码片段及其等价的类型声明。这个示例展示了它在做什么，以及它使用了哪些语言特性：

TypeScript实现代码（.ts） ​

typescript
import {Subscriber} from './Subscriber'
import {Subscription} from './Subscription'
import {PartialObserver, Subscribable, TeardownLogic} from './types'

export class Observable<T> implements Subscribable<T> {
  public _isScalar: boolean = false
  constructor(
    subscribe?: (
      this: Observable<T>,
      subscriber: Subscriber<T>
    ) => TeardownLogic
  ) {
    if (subscribe) {
      this._subscribe = subscribe
    }
  }
  
  static create<T>(subscribe?: (subscriber: Subscriber<T>) => TeardownLogic) {
    return new Observable<T>(subscribe)
  }
  
  subscribe(observer?: PartialObserver<T> | ((value: T) => void),
           error?: (error: any) => void,
           complete?: () => void
  ): Subscription {
    // 实现逻辑
  }
}

对应的类型声明（.d.ts） ​

typescript
import {Subscriber} from './Subscriber'
import {Subscription} from './Subscription'  
import {PartialObserver, Subscribable, TeardownLogic} from './types'

export declare class Observable<T> implements Subscribable<T> {
  public _isScalar: boolean
  constructor(
    subscribe?: (
      this: Observable<T>,
      subscriber: Subscriber<T>
    ) => TeardownLogic
  )
  
  static create<T>(subscribe?: (subscriber: Subscriber<T>) => TeardownLogic): Observable<T>
  
  subscribe(observer?: PartialObserver<T> | ((value: T) => void),
           error?: (error: any) => void,
           complete?: () => void
  ): Subscription
}

类型声明与实现的区别 ​

主要差异点： ​

  1. 使用 declare 关键字：在类型声明中，我们在类前面加上 declare 关键字

  2. 移除所有实现：

    • ❌ 删除属性的初始值：boolean = false → boolean
    • ❌ 删除构造函数体和方法体
    • ❌ 删除所有实际的实现代码

  3. 保留类型信息：

    • ✅ 保留所有类型注解
    • ✅ 保留方法签名
    • ✅ 保留接口实现关系
    • ✅ 保留泛型参数

  4. 添加返回类型：

    • 为静态方法明确添加返回类型
    • 为实例方法明确添加返回类型

类型声明的作用 ​

类型声明文件的主要作用是：

  1. 类型安全：为JavaScript库提供TypeScript类型检查
  2. 智能提示：在IDE中获得自动完成和参数提示
  3. 文档作用：作为API的类型文档
  4. 渐进迁移：允许在迁移过程中为JavaScript代码添加类型

何时使用类型声明 ​

你会在以下情况下使用类型声明：

  1. 为第三方JavaScript库添加类型
  2. 为现有JavaScript代码逐步添加类型
  3. 为全局变量或库提供类型定义
  4. 在迁移过程中作为过渡方案

如何编写类型声明文件 ​

使用 declare 关键字 ​

注意在类前面使用 declare 关键字。我们不能在类型声明中实际定义一个类，但可以声明我们在 .d.ts 文件对应的JavaScript文件中定义了一个类。

因为类型声明不包含实现，我们只保留两个重载：在其他类型声明中声明的内容，以及和这个声明对应的JavaScript代码中的内容。

运行类型声明文件 ​

使用启用了 declarations 标志的TSC来运行这段代码（tsc -d），会产生以下Observable.d.ts类型声明：

typescript
import { Subscriber } from './Subscriber'
import { Subscription } from './Subscription'
import { PartialObserver, Subscribable, TeardownLogic } from './types'

export declare class Observable<T> implements Subscribable<T> {
    _isScalar: boolean
    constructor(subscribe?: (this: Observable<T>, subscriber: Subscriber<T>) => TeardownLogic)
    static create<T>(subscribe?: (subscriber: Subscriber<T>) => TeardownLogic): Observable<T>
    subscribe(observer?: PartialObserver<T> | ((value: T) => void), 
             error?: (error: any) => void, 
             complete?: () => void): Subscription
}

类型声明的优化效果 ​

想想如果RxJS的作者想要在NPM上打包类型信息，他们的TypeScript版本应用需要做什么。对于RxJS库，大约有两个选择：

  1. 打包TypeScript源代码：用户需要自己为对应的.ts文件编译TypeScript应用。但这个让TypeScript版本开发者变慢，它也必须重新配置你的TypeScript编译器

  2. 代码编辑器支持：支持TypeScript的代码编辑器（如VSCode）会读取这些.d.ts文件来给你内置的类型提示和高亮显示，以显示他们不会检查TypeScript

  3. 加快编译速度：通过避免不必要地重新编译你的TypeScript代码，显著地加快编译时间

类型声明是告诉TypeScript的一种方式："这里存在这个在JavaScript中定义的东西。"这就是关于类型声明的全部——我们称之为环境，以区别于包含值的常规声明；环境变量声明是在没有使用 declare 关键字的环境中定义的常规声明。

类型声明的几种用途 ​

你可以将类型声明用于几个方面：

1. 声明全局变量 ​

当你从全局定义的JavaScript库使用一个全局变量，或者需要在你的项目中去使用一个环境模块声明：

typescript
// 声明全局可用的jQuery
declare var $: JQueryStatic

2. 定义全局类型 ​

要定义一个全局可用的类型，或者要描述一个在你的环境中可用但没有ambient模块声明的类型：

typescript
// 声明全局可用的类型
declare type UserID = string

3. 告诉TypeScript关于第三方模块 ​

要告诉TypeScript关于你通过NPM安装的第三方模块（使用ambient模块声明）：

typescript
// 声明第三方模块
declare module 'some-third-party-module' {
  export function doSomething(): void
}

Ambient模块声明 ​

ambient模块声明是一种特殊的类型声明，用于描述通过模块系统可用的JavaScript代码。比如说，如果你想要为Promise.catch添加过程，你可以在.d.ts文件中创建一个ambient模块声明：

typescript
declare module 'my-promise-library' {
  export interface MyPromise<T> {
    catch<U>(onRejected: (reason: any) => U | PromiseLike<U>): MyPromise<U>
  }
}

这告诉TypeScript："存在一个可以通过 import 或 require 访问的模块，它导出这些类型和值。"

类型声明文件的命名规则 ​

类型声明文件的命名有一定的约定，无论你将其用于script模式还是module模式：

Script模式 vs Module模式 ​

根据早期讨论的模块约定，我们给文件一个.d.ts扩展名，如果它有对应的.js文件；否则，我们使用.d.ts扩展名。文件命名约定是为了坚持一个简单的顶级逻辑：d.ts意味着"这包含数据类型定义"，而单个类型声明文件可以包含任意多个类型声明。

文件扩展名规则 ​

  • 有对应JavaScript文件：使用.d.ts扩展名
  • 纯类型定义文件：也使用.d.ts扩展名
  • 命名约定：通常与对应的JavaScript文件同名

环境变量声明（Ambient Variable Declarations） ​

环境变量声明用来告诉TypeScript关于可以在任何.ts或.js文件中使用的全局变量，而无需显式导入。

基本语法 ​

让我们通过一个栈溢出错误的例子来看看。假设你正在运行一个Node.js程序，但得到导入检查：process.env.NODE_ENV（这是"development"或"production"）在某个地方。当程序运行时，你得到一个错误的运行时错误：

bash
Uncaught ReferenceError: process is not defined.

声明全局变量 ​

你声明给TypeScript的是有一个全局对象process，它有一个简单的属性env。一旦你告诉TypeScript关于这个，错误就消失了，你可以安全地定义你的 process 全局。

typescript
declare var process: {
  env: {
    NODE_ENV: 'development' | 'production'
  }
}

你现在声明给TypeScript的是有一个全局对象process，它有一个简单的属性env.NODE_ENV。一旦告诉TypeScript这些，你就可以安全地定义你的process全局变量了。

在代码中使用 ​

typescript
// 现在可以安全使用了
process = {
  env: {
    NODE_ENV: 'production'
  }
}

TypeScript不会抱怨，你可以从错误中清理你存在的window全局。

但在这种情况下，TypeScript正在保护你。你真的想要扩展window，你想要安全地做到这一点。

TypeScript配置：lib设置 ​

TypeScript包含一套内置描述JavaScript标准库的类型声明，但不包括内置JavaScript类型，如数组和Promise，以及在浏览器中可用的方法如window对象。

什么是lib设置 ​

你可以通过你的tsconfig.json的lib字段来控制TypeScript引入哪些类型声明。跳转到"lib"是deep dive到你项目的lib设置的步骤链接。

lib设置的作用 ​

json
{
  "compilerOptions": {
    "lib": ["es2020", "dom"]
  }
}

这个配置告诉TypeScript包含：

  • ES2020：ECMAScript 2020的类型定义
  • DOM：浏览器DOM API的类型定义

环境类型声明（Ambient Type Declarations） ​

环境类型声明遵循与环境变量声明相同的规则：声明在.d.ts文件中，它会在全局可用你项目中的其他文件，而无需显式导入。

基本示例 ​

比如说，让我们声明一个全局可用的工具类型：

typescript
// types.d.ts
type UserID = string

高级示例 ​

你可以声明更复杂的类型：

typescript
// globals.d.ts  
type APIResponse<T> = {
  data: T
  status: number
  message: string
}

interface Window {
  myGlobalFunction: () => void
}

使用环境类型 ​

现在你可以在项目的任何地方使用这个类型，而无需显式导入：

typescript
// 在任何文件中都可以直接使用
let userId: UserID = "user123"
let response: APIResponse<User> = await fetchUser()

// 现在window对象有了类型安全的myGlobalFunction
window.myGlobalFunction()

实际应用建议 ​

考虑使用环境类型声明来建模在你整个应用程序中使用的数据类型。例如，你可以使用它们来使我们在"模拟标称类型"中开发的UserID类型全局可用：

typescript
// globals.d.ts
type UserID = string & { readonly brand: unique symbol }

function toArray<T>(a: T): ToArray<T> {
  // ...
}

现在，你可以在应用程序的任何地方使用UserID，而无需明确导入它。

环境模块声明（Ambient Module Declarations） ​

当你使用JavaScript模块并想要快速声明一些类型以便你可以安全地使用它时——无需向JavaScript模块的GitHub仓库或DefinitelyTyped贡献类型声明——环境模块声明是你要使用的工具。

基本语法 ​

环境模块声明是常规类型声明，被特殊的declare module语法包围：

typescript
declare module 'module-name' {
  export type MyType = number
  export type MyDefaultType = (a: string) => string
  export let myExport: MyType
  export default myDefaultExport: MyDefaultType
}

实际使用示例 ​

模块名称（在这个例子中是'module-name'）对应于确切的导入路径。当你导入该路径时，你的环境模块声明告诉TypeScript什么是可用的：

typescript
import ModuleName from 'module-name'
// ModuleName 现在有类型 MyDefaultType

import { myExport } from 'module-name'
// myExport 现在有类型 MyType

嵌套模块声明 ​

如果你有嵌套模块，确保在声明中包含整个导入路径：

typescript
declare module '@most/core' {
  // 类型声明
}

快速类型声明 ​

如果你只想快速告诉TypeScript"我正在导入这个模块——我稍后会类型化它，现在就假设它是any"，保留头部但省略实际声明：

typescript
// 声明一个可以导入的模块，其中每个导入都是any
declare module 'unsafe-module-name'

现在如果你使用这个模块，它就不那么安全了：

typescript
import {x} from 'unsafe-module-name'
x // any

通配符导入支持 ​

模块声明支持通配符导入，所以你可以为任何匹配给定模式的导入路径提供类型。使用通配符（*）来匹配导入路径：

typescript
// 为使用webpack的json-loader导入的JSON文件提供类型
declare module 'json!*' {
  let value: object
  export default value
}

// 为使用webpack的style-loader导入的CSS文件提供类型  
declare module '*.css' {
  let css: CSSRuleList
  export default css
}

现在，你可以加载JSON和CSS文件：

typescript
import a from 'json!myFile'
a // object

import b from './widget.css'
b // CSSRuleList

注意事项 ​

重要：为了让最后两个示例工作，你需要配置你的构建系统来加载json和css文件。你可以向TypeScript声明这些路径模式是安全导入的，但TypeScript无法自己构建它们。

实际项目中的应用场景 ​

1. 第三方库快速集成 ​

当你需要快速使用一个没有类型定义的JavaScript库时：

typescript
declare module 'some-library' {
  export function doSomething(param: string): number
  export default class SomeClass {
    constructor(options: any)
    method(): void
  }
}

2. 构建工具集成 ​

为webpack、rollup等构建工具的特殊导入提供类型：

typescript
// 为图片导入提供类型
declare module '*.png' {
  const value: string
  export default value
}

declare module '*.svg' {
  const value: string
  export default value
}

3. 渐进迁移 ​

在从JavaScript迁移到TypeScript的过程中，临时为模块提供类型：

typescript
// 暂时声明为any，等待后续完善
declare module './legacy-module'

// 或者提供基本的类型信息
declare module './utils' {
  export function helper(input: string): string
}

从 JavaScript 逐步迁移到 TypeScript ​

TypeScript从设计之初就考虑了与JavaScript的互操作性。这意味着在迁移过程中，你可以在每个步骤都获得益处，从无类型代码逐步过渡到严格类型检查的代码。

迁移策略总览 ​

迁移策略概览 ​

在TypeScript中从既有JavaScript代码库受益的理想结果是：你的代码库应该完全用TypeScript编写。为了实现这个目标，你可以在仍有大量JavaScript代码的情况下从中受益。TypeScript本身有助于消除bug并让你有更多时间进行推理。

一些能在编译时抓住的错误示例：

  • 拼写错误：变量名、方法名的错误
  • 类型错误：向函数传递错误类型的参数
  • 引用错误：使用未定义的变量或方法
  • 逻辑错误：一些可以通过类型检查发现的逻辑问题

迁移的基本步骤 ​

迁移的具体步骤 ​

此迁移过程包括以下主要步骤：

  1. 添加TSC到你的项目
  2. 开始为现有的JavaScript代码进行类型检查
  3. 一次迁移一个JavaScript文件到TypeScript
  4. 为你的依赖项安装类型声明，或者自己编写类型声明，或者为没有类型或编写类型声明的依赖项贡献给DefinitelyTyped
  5. 启用严格的TSC设置

这个过程需要一定时间，但你将在每个步骤都获得安全和效率上的收益，并随着进程的推进发现更多好处。让我们逐步了解每个步骤。

步骤1：添加TSC ​

在混合TypeScript和JavaScript的代码库中工作时，首先要让TSC编译你的TypeScript文件。在你的tsconfig.json中启用：

json
{
  "compilerOptions": {
    "allowJs": true
  }
}

通过这一个改变，你现在可以使用TSC来编译你的JavaScript。只需在你的构建过程中运行TSC，或者继续运行你现有的JavaScript构建工具。

allowJs选项的作用 ​

启用allowJs后，TypeScript编译器将：

  • 处理.js文件以及.ts文件
  • 对JavaScript文件进行基本的语法检查
  • 允许从TypeScript文件导入JavaScript模块
  • 保持现有的构建流程

步骤2：为JavaScript启用类型检查（可选） ​

现在TSC正在处理你的JavaScript，为什么不让它类型检查呢？虽然你可能还没有在你的JavaScript中添加显式类型注解，但你可能会记住TypeScript在推断许多类型方面有多么好。启用你的tsconfig.json中的这个功能：

json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": true
  }
}

checkJs的效果 ​

现在，无论何时TypeScript编译JavaScript文件，它都会尽力推断JavaScript代码的类型并进行类型检查。如果一次性的类型检查报告的错误太多，你可以通过以下方式逐个文件启用：

如果你的代码库很大且启用checkJs报告太多类型错误，可以通过以下方式关闭：

json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": false
  }
}

然后在JavaScript文件顶部添加类型检查：

javascript
// @ts-check
// 这个文件现在会进行类型检查

function add(a, b) {
  return a + b  // TypeScript会推断类型
}

类型推断的威力 ​

即使在JavaScript文件中，TypeScript也能提供强大的类型推断：

javascript
// @ts-check

// TypeScript推断这些类型
let name = "John"           // string
let age = 25               // number  
let users = []             // any[]

// 函数参数和返回类型推断
function greet(userName) {   // userName: any
  return "Hello " + userName // return: string
}

// 对象类型推断
let user = {
  name: "Alice",    // string
  age: 30          // number
}

重要注意事项 ​

注意：由于TypeScript无法推断所有内容（例如函数参数类型），它会对无法推断的内容使用any类型。记住any实际上关闭了类型检查，所以在迁移过程中遇到any类型时可以暂时忽略。在tsconfig.json中：

json
{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": true,
    "noImplicitAny": false
  }
}

当你迁移到TypeScript时，不要忘记重新启用noImplicitAny！它可能会发现一大批你遗漏的代码错误。当然，你对熟悉的地方比较了熟悉。除了TypeScript之外，你能够在JavaScript中指定类型，使用TypeScript能够理解的特殊语法。

JavaScript类型推断的工作原理 ​

当TypeScript运行你的JavaScript代码时，它使用一个名为inference algorithm的东西来为TypeScript代码。具体来说：

1. 所有函数参数推断为any ​

javascript
// TypeScript推断的类型
function add(a, b) {  // a: any, b: any
  return a + b        // return: any
}

2. 属性和函数的类型从使用中推断（而不是从声明中推断） ​

javascript
class A {
  constructor() {
    this.number = 1         // number: any (从使用推断)
    this.method() {         // 从使用推断 method()
      return 'foo'
    }
    this.otherMethod() {
      return ['array', 'of', 'strings']  // 从使用推断返回string[]
    }
  }
}

3. 声明对象、类或函数后，你可以为其分配额外的属性 ​

在字面上，TypeScript通过为每个类和函数声明生成对应的命名空间来实现这一点，并将每个对象字面量自动转换为每个对象字面量：

javascript
// 可以为函数添加属性
function myFunc() {
  return 'hello'
}
myFunc.someProperty = 42    // 这在JavaScript中是合法的

步骤2b：添加JSDoc注释（可选） ​

也许你正在匆忙中，只需要为添加到旧JavaScript文件的新函数添加简单类型注释。你要一个函数，该函数将单词替换为另一个单词：

javascript
export function toPascalCase() {
  return word.replace(/(\w)/, (_, c) => c.toUpperCase()).join('').toLowerCase()
}

如果不将unit转换为TypeScript，就完全不能——你仍然可以通过一些巧妙的JSDoc注释来获得相当不错的类型推断：

javascript
/**
 * @param word {string} An input string
 * @returns {string} The string in PascalCase  
 */
export function toPascalCase(word) {
  return word.replace(/(\w)/, (_, c) => c.toUpperCase()).join('').toLowerCase()
}

JSDoc注释的效果 ​

没有JSDoc注释，TypeScript会将toPascalCase的类型推断为： (word: any) => string，当TypeScript编译你的代码时它会知道这一点的类型。

使用JSDoc注释后，你得到了更多类型文档： (word: string) => string。这样，你得到更好的类型安全，而无需完全迁移到TypeScript！

支持的JSDoc注释 ​

以下是一些TypeScript支持的JSDoc注释：

javascript
/**
 * @param {string} name - 用户名称
 * @param {number} age - 用户年龄  
 * @param {boolean} [isActive] - 是否激活（可选参数）
 * @returns {string} 格式化的用户信息
 */
function formatUser(name, age, isActive) {
  // 实现
}

/**
 * @param {string|number} value - 可以是字符串或数字
 * @returns {string} 转换后的字符串
 */
function toString(value) {
  return String(value)
}

/**
 * @param {Array<string>} items - 字符串数组
 * @returns {string} 连接后的字符串
 */
function joinItems(items) {
  return items.join(', ')
}

虽然JSDoc提供了基本的类型支持，但TypeScript提供了更强大的功能：

特性JSDocTypeScript

步骤3：将文件重命名为.ts ​

一旦你将TSC添加到构建过程中，并可选地开始对JavaScript代码进行类型检查和注释，现在是时候将文件迁移到TypeScript了。

逐个文件迁移策略 ​

一次一个文件，将你的文件扩展名从.js（或.coffee、.es6等）更改为.ts。一旦你这样做了，TypeScript就会为你抛出各种有用的编译错误，包括来自名称拼写错误到错误类型的函数调用等等。

迁移的基本步骤 ​

  1. 重命名文件：将.js文件重命名为.ts
  2. 修复类型错误：逐个解决TypeScript报告的类型错误
  3. 添加类型注解：为函数参数、返回值等添加明确的类型
  4. 测试验证：确保重构后的代码功能正常

处理类型错误的策略 ​

有两种主要的处理方式：

方式1：快速但粗糙的方法 ​

使用带有类型注解的ambient类型声明来定义所有包含它们的文件。如果你启用了checkJs，你可以在TypeScript中执行相同的操作，但你希望继续前进：

typescript
// @ts-ignore
let result = someComplexFunction()

// 或者使用any类型快速跳过
let data: any = complexJavaScriptCode()

方式2：严格但安全的方法 ​

手动添加类型注解，修复所有类型错误：

typescript
// 之前的JavaScript
function processUser(user) {
  return {
    id: user.id,
    name: user.name.toUpperCase(),
    email: user.email || 'no-email'
  }
}

// 迁移后的TypeScript
interface User {
  id: string
  name: string
  email?: string
}

interface ProcessedUser {
  id: string
  name: string
  email: string
}

function processUser(user: User): ProcessedUser {
  return {
    id: user.id,
    name: user.name.toUpperCase(),
    email: user.email || 'no-email'
  }
}

选择合适的迁移方式 ​

两种做法都是公平的游戏，这取决于你想要进行的。因为大多数迁移应该逐渐运行，强制所有没有类型的代码是暴露所有错误之后重新命名。类型错误在重构方法的意图当中无法预见到，你失踪了一些我们从类型检查中获得的最大好处是实时类型。

重要提示 ​

小贴士：如果你选择进行快速而粗糙的路线，一个有用的技巧是定义一个环境类型别名用于any，并且使用它代替在你的类型检查器中的any，以使你的类型暴露给追踪缺失类型。你也可以称之为更具体的东西，比如用它来获取名称类型搜索。

typescript
// globals.d.ts
type TODO_FIXME_ADD_TYPES = any

// 使用时
function processData(input: TODO_FIXME_ADD_TYPES): TODO_FIXME_ADD_TYPES {
  // TODO: 添加具体类型
  return input
}

这样你可以轻松搜索TODO_FIXME_ADD_TYPES来找到所有需要添加类型的地方。

步骤4：让它更严格 ​

一旦你将关键的JavaScript代码迁移到TypeScript，你会想要让TypeScript更严格。参见附录F以获得完整的严格标志列表。

推荐的严格设置 ​

最终，你可以通过启用TSC的JavaScript互操作标志来禁用所有这些，强制所有代码都以严格类型的TypeScript编写：

json
{
  "compilerOptions": {
    "allowJs": false,
    "checkJs": false,
    "strict": true
  }
}

strict模式包含的检查 ​

启用strict: true等同于启用以下所有严格检查：

  • noImplicitAny: 禁止隐式any类型
  • strictNullChecks: 严格的null检查
  • strictFunctionTypes: 严格的函数类型检查
  • strictBindCallApply: 严格的bind/call/apply检查
  • strictPropertyInitialization: 严格的属性初始化检查
  • noImplicitReturns: 确保函数所有路径都有返回值
  • noImplicitThis: 禁止隐式的this类型

逐步启用严格模式 ​

你可以逐步启用这些设置：

json
{
  "compilerOptions": {
    "strict": false,
    "noImplicitAny": true,
    "strictNullChecks": true
    // 逐步添加其他严格检查
  }
}

这将在final round中清除类型相关错误。修复这些错误，你就完成了与严格类型的TypeScript世界精确相关的完整迁移！

JavaScript的TypeScript类型查找 ​

当你从TypeScript文件导入JavaScript文件时，TypeScript遵循一个算法来查找你的JavaScript代码的类型声明。记住"文件"和"TypeScript编程入门"从算法：

类型查找算法 ​

  1. 查找同名.d.ts文件：寻找与你的.js文件同名的.d.ts文件。如果存在，将其用作该.js文件的类型声明。

例如，如果你有：

utils.js
utils.d.ts  // TypeScript会使用这个文件作为utils.js的类型声明

完整的类型查找过程 ​

让我们通过一个具体示例来理解TypeScript的类型查找算法：

本地文件的类型查找 ​

假设你有以下文件夹结构：

my-app/
├─src/
│ ├─index.ts
│ └─legacy/
│   ├─old-file.js
│   └─old-file.d.ts

当你从index.ts导入old-file时：

typescript
// index.ts
import './legacy/old-file'

TypeScript会使用src/legacy/old-file.d.ts作为./legacy/old-file的类型声明来源。

其他查找策略 ​

如果找不到同名的.d.ts文件，TypeScript会依次尝试：

  1. 如果启用了allowJs和checkJs：推断.js文件的类型（基于JSDoc注释）
  2. 否则：将整个模块视为any类型

第三方模块的类型查找 ​

当导入第三方JavaScript模块（即从node_modules安装的NPM包）时，TypeScript使用稍微不同的算法：

1. 查找本地类型声明 ​

首先在应用的文件夹结构中查找本地类型声明：

my-app/
├─node_modules/
│ └─foo/
├─src/
│ ├─index.ts
│ └─types.d.ts

如果types.d.ts包含：

typescript
declare module 'foo' {
  let bar: {}
  export default bar
}

当你导入foo时，TypeScript会使用types.d.ts中的环境模块声明作为foo的类型来源：

typescript
// index.ts
import bar from 'foo'  // 使用types.d.ts中的声明

2. 查找package.json中的类型字段 ​

如果模块的package.json定义了types或typings字段，使用该字段指向的.d.ts文件作为模块的类型声明来源：

json
{
  "name": "some-module",
  "main": "./lib/index.js",
  "types": "./lib/index.d.ts"
}

3. 查找@types包 ​

否则，遍历目录查找node_modules/@types目录，该目录包含模块的类型声明。

例如，如果你安装了React：

bash
npm install react --save
npm install @types/react --save-dev

文件结构会是：

my-app/
├─node_modules/
│ ├─@types/
│ │ └─react/
│ │   └─index.d.ts
│ └─react/
│   └─index.js
├─src/
│ └─index.ts

当你导入React时，TypeScript会找到@types/react文件夹并使用它作为React的类型声明来源：

typescript
// index.ts
import * as React from 'react'  // 使用@types/react中的类型

4. 回退到本地查找算法 ​

如果以上都失败，则回到步骤1-3的本地查找算法。

实际应用示例 ​

为第三方库添加类型 ​

如果某个库没有类型声明，你可以：

方法1：创建本地声明文件

typescript

declare module 'some-library' {
  export function doSomething(param: string): number
  export default class SomeClass {
    constructor(options: any)
    method(): void
  }
}

方法2：贡献到@types 为DefinitelyTyped项目贡献类型声明，让所有人受益。

方法3：使用快速声明

typescript
// 临时解决方案
declare module 'some-library'  // 所有导入都是any类型

类型查找的最佳实践 ​

  1. 优先使用官方类型：检查库是否已包含TypeScript类型
  2. 查找@types包：在DefinitelyTyped中寻找社区维护的类型
  3. 创建本地声明：为小型或特定的库创建本地类型声明
  4. 贡献类型：考虑为缺少类型的流行库贡献类型声明
  5. 渐进完善：先使用基本类型，随时间逐步完善

TSC设置：types和typeRoots ​

默认查找行为 ​

默认情况下，TypeScript编译器会自动在项目根目录的node_modules/@types文件夹以及其上级目录中查找类型声明。

覆盖默认行为 ​

要覆盖这种默认行为以进行全局类型声明，配置tsconfig.json中的typeRoots字段：

json
{
  "compilerOptions": {
    "typeRoots": ["./node_modules/@types"]
  }
}

这告诉TypeScript去查找类型声明的地方。也可以包含多个路径：

json
{
  "compilerOptions": {
    "typeRoots": ["./types", "./node_modules/@types"]
  }
}

精确控制类型导入 ​

为了更精确的控制，使用types选项在你的tsconfig.json中指定确切你想要引入哪些包：

json
{
  "compilerOptions": {
    "types": ["react"]
  }
}

这只会加载@types/react包，忽略其他所有的@types包。

实际配置示例 ​

json
{
  "compilerOptions": {
    "typeRoots": ["./types", "./node_modules/@types"],
    "types": ["node", "react", "jest"]
  }
}

安装第三方JavaScript的三种情况 ​

在这种情况下，你有几种选择：

选择1：使用ambient模块声明快速解决

三种处理策略 ​

对于没有类型声明的第三方JavaScript，你有三个主要选择：

declare module 'some-module' {

创建高质量的类型声明 ​

编写第三方JavaScript的类型声明是一项复杂的任务，但以下是一些基本步骤：

typescript
// 为一个假想的工具库创建类型声明
declare module 'awesome-utils' {
  // 导出的函数
  export function debounce<T extends (...args: any[]) => any>(
    func: T, 
    wait: number
  ): T
  
  export function throttle<T extends (...args: any[]) => any>(
    func: T, 
    wait: number
  ): T
  
  // 导出的类
  export class EventEmitter {
    on(event: string, listener: (...args: any[]) => void): this
    emit(event: string, ...args: any[]): boolean
    off(event: string, listener?: (...args: any[]) => void): this
  }
  
  // 导出的接口
  export interface UtilOptions {
    timeout?: number
    retries?: number
  }
  
  // 默认导出
  export default function utils(options?: UtilOptions): {
    version: string
    ready: boolean
  }
}

现在使用导入 ​

现在当你从 'awesome-utils' 导入 debounce 时，TypeScript会知道确切的 debounce 类型。它不再是any，而是 <T extends (...args: any[]) => any>(func: T, wait: number) => T：

typescript
import { debounce, EventEmitter } from 'awesome-utils'

// 完整的类型支持
const debouncedFn = debounce((x: number) => x * 2, 100)  
// debouncedFn: (x: number) => number

const emitter = new EventEmitter()
emitter.on('data', (data: any) => console.log(data))

这样编写的类型声明既提供了类型安全，又保持了与JavaScript原生行为的一致性。

自动生成类型声明的工具 ​

注意：为无类型JavaScript自动生成类型声明是一个活跃的研究领域。可以查看 dts-gen 工具，它提供了一种为任何第三方JavaScript模块自动生成类型声明脚手架的方法。

配置说明 ​

  1. 导入无类型JavaScript

    • 最简单的集成方式
    • 所有JavaScript导入被视为any类型
    • 类型安全性最低

  2. 导入并检查JavaScript

    • 启用基本的JavaScript类型检查
    • TypeScript尝试推断JavaScript代码的类型
    • 提供中等程度的类型安全

  3. 导入并检查JSDoc注释的JavaScript

    • 利用JSDoc注释提供类型信息
    • 在严格模式下提供良好的类型安全
    • 作为迁移过程中的过渡方案

  4. 导入带有类型声明的JavaScript

    • 为JavaScript代码编写.d.ts类型声明文件
    • 提供完整的类型安全而不修改JavaScript源代码
    • 适合无法修改的第三方库

  5. 导入TypeScript

    • 最理想的状态
    • 完整的类型安全和开发体验
    • 长期目标

总结 ​

第11章深入探讨了TypeScript与JavaScript的互操作性，这是现实项目中不可避免的重要话题。通过本章学习，我们掌握了：

核心概念 ​

  • 类型声明文件：使用.d.ts文件为JavaScript代码提供类型信息
  • 环境声明：声明全局变量、类型和模块的存在
  • 渐进迁移：从JavaScript逐步迁移到TypeScript的策略

实用技能 ​

  • 编写和使用类型声明文件
  • 配置TypeScript编译器处理JavaScript代码
  • 使用JSDoc注释提供过渡期类型支持
  • 管理第三方库的类型声明

迁移策略 ​

  1. 添加TSC：启用allowJs配置
  2. 类型检查：启用checkJs进行JavaScript类型检查
  3. 逐步迁移：将.js文件重命名为.ts并修复类型错误
  4. 严格模式：最终启用strict模式获得完整类型安全

类型查找机制 ​

TypeScript有完整的类型查找算法，能够自动找到本地类型声明、@types包和package.json中定义的类型入口。

重要注意事项 ​

  1. 通配符匹配：通配符匹配使用 * 遵循与常规glob模式匹配相同的规则
  2. DefinitelyTyped：这是JavaScript类型声明的开源仓库，是社区维护类型声明的重要资源
  3. 性能考虑：对于大型项目，可能需要通过TSC对每个单独文件进行类型检查会比较慢，可以参考"项目引用"来提高大型项目的性能
  4. 模块模式 vs 脚本模式：严格来说，某些特性仅适用于模块模式而非脚本模式文件

实际应用指导 ​

与JavaScript的互操作可能是TypeScript最棘手的方面之一；通过本章介绍的所有工具，你现在已经具备了在自己项目中实现互操作的能力。

从无类型JavaScript到完全类型化TypeScript的迁移路径：

  1. 开始时允许JavaScript文件存在
  2. 逐步添加类型检查
  3. 为关键模块编写类型声明
  4. 逐个文件迁移到TypeScript
  5. 最终实现完全的类型安全

通过掌握这些互操作技能，我们能够在现实世界的混合代码库中成功引入和使用TypeScript，享受类型安全带来的好处，同时保持与JavaScript生态系统的兼容性。
