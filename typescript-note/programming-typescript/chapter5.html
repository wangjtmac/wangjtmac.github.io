第5章：类和接口 ​

章节概述 ​

对于来自面向对象编程语言的程序员，类是组织和思考代码的基本工具，是封装的主要单位。TypeScript 类借鉴了 C# 的许多特性，支持可见性修饰符、属性初始化器、装饰器和接口等功能。由于 TypeScript 类编译为常规 JavaScript 类，你也可以在 TypeScript 中表达 JavaScript 习惯用法如 mixins。

TypeScript 的一些类特性（如属性初始化器和装饰器）由 JavaScript 类支持，并生成运行时代码。其他特性（如可见性修饰符、接口和泛型）是 TypeScript 独有的，仅在编译时存在，编译应用程序到 JavaScript 时不会生成任何代码。

本章将通过一个扩展的国际象棋引擎示例，指导你在 TypeScript 中使用类，让你不仅能直观了解 TypeScript 的面向对象特性，还能了解我们使用它们的方式和原因。

类和继承 ​

我们将构建一个国际象棋引擎，为两个玩家轮流下棋提供 API。

首先勾勒出类型结构：

typescript
// 表示一个国际象棋游戏
class Game {}

// 表示一个棋子
class Piece {}

// 表示棋子位置的坐标集合
class Position {}

国际象棋坐标系统 ​

国际象棋使用标准代数记号表示位置：

  • 文件（Files）：A-H 列（从左到右）
  • 等级（Ranks）：1-8 行（从下到上）
  • 每个位置由文件和等级组成，如 "E4"

基本类型定义 ​

让我们为 Piece 类添加颜色和位置：

typescript
type Color = 'Black' | 'White'
type File = 'A' | 'B' | 'C' | 'D' | 'E' | 'F' | 'G' | 'H'
type Rank = 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8

class Position {
  constructor(
    private file: File,
    private rank: Rank
  ) {}
}

棋子继承层次 ​

国际象棋有六种棋子类型：

typescript
// 基类
class Piece {}

// 具体棋子类型
class King extends Piece {}
class Queen extends Piece {}
class Bishop extends Piece {}
class Knight extends Piece {}
class Rook extends Piece {}
class Pawn extends Piece {}

### 完整的 Piece 类定义

```typescript
class Piece {
  protected position: Position
  constructor(
    private readonly color: Color,
    file: File,
    rank: Rank
  ) {
    this.position = new Position(file, rank)
  }
}

关键点说明：

  • 由于颜色、等级和文件相对较少，我们可以手动枚举它们的可能值作为类型字面量
  • private 访问修饰符在构造函数中自动将参数分配给 this，并设置其可见性为私有
  • 我们将实例变量 position 声明为 protected，这样 Piece 的实例和子类都可以访问它，但 Piece 实例外的代码不能访问

访问修饰符 ​

TypeScript 支持三种类属性和方法的访问修饰符：

public ​

  • 可从任何地方访问，这是默认访问级别

protected ​

  • 可从此类的实例及其子类访问

private ​

  • 仅可从此类的实例访问

使用访问修饰符，你可以设计不会暴露过多信息的类，而是暴露定义良好的 API 供其他人使用。

我们定义了一个 Piece 类，但我们不希望用户直接实例化新的 Piece —— 我们希望他们扩展 Piece，因此我们可以使用类型系统通过 abstract 关键字强制执行：

typescript
// 抽象类 Piece
abstract class Piece {
  protected position: Position
  constructor(
    private readonly color: Color,
    file: File,
    rank: Rank
  ) {
    this.position = new Position(file, rank)
  }

  // 抽象方法，子类必须实现
  abstract canMoveTo(position: Position): boolean
}

现在如果你尝试直接实例化 Piece，TypeScript 会报错：

typescript
new Piece('White', 'E', 1) // 错误 TS2511: 无法创建抽象类的实例

abstract 关键字意味着你不能直接实例化该类，但这并不意味着你不能在其上定义方法。

我们的 Piece 类现在：

  • 告诉其子类它们必须实现一个名为 canMoveTo 的方法，该方法与给定签名兼容
  • 如果一个类扩展 Piece 但忘记实现 canMoveTo 方法，TypeScript 会报错

TSC 标志：严格空值检查和严格属性初始化 ​

启用严格检查 ​

要启用类实例变量的明确赋值检查和严格属性初始化，请在 tsconfig.json 中使用 strictNullChecks 和 strictPropertyInitialization 标志：

json
{
  "compilerOptions": {
    "strictNullChecks": true,
    "strictPropertyInitialization": true
  }
}

如果你已经在使用 strict 标志，那么你已经启用了这些检查。

严格属性初始化的影响 ​

启用这些标志后，TypeScript 要求你在构造函数中明确初始化所有属性，或者在属性声明时提供初始值。这有助于防止运行时错误，确保所有属性都有明确的值。

具体实现示例 ​

Position 类的距离计算 ​

typescript
class Position {
  constructor(
    private file: File,
    private rank: Rank
  ) {}

  distanceFrom(position: Position) {
    return {
      rank: Math.abs(position.rank - this.rank),
      file: Math.abs(position.file.charCodeAt(0) - this.file.charCodeAt(0))
    }
  }
}

King 类的实现 ​

typescript
class King extends Piece {
  canMoveTo(position: Position) {
    let distance = this.position.distanceFrom(position)
    return distance.rank < 2 && distance.file < 2
  }
}

游戏初始化 ​

当我们创建新游戏时，会自动创建棋盘和一些棋子：

typescript
class Game {
  private pieces = Game.makePieces()

  private static makePieces() {
    return [
      // Kings
      new King('White', 'E', 1),
      new King('Black', 'E', 8),

      // Queens
      new Queen('White', 'D', 1),
      new Queen('Black', 'D', 8),

      // Bishops
      new Bishop('White', 'C', 1),
      new Bishop('White', 'F', 1),
      new Bishop('Black', 'C', 8),
      new Bishop('Black', 'F', 8),

      // 更多棋子...
    ]
  }
}

类型安全的文件和等级 ​

由于我们严格类型化了 Rank 和 File，如果我们输入了另一个字母（如 'J'）或超出范围的数字（如 5-2），TypeScript 会在编译时给出错误（图 5-2）。

typescript
return [
  // 错误：类型 '"3"' 的参数不能赋值给类型 'File' 的参数
  new King('White', '10', 1),
  // 错误：类型 '3' 的参数不能赋值给类型 'Rank' 的参数
  new King('Black', 'J', 12),
]

这足以展示 TypeScript 类如何工作——我将避免深入了解游戏的细节，如如何知道骑士何时可以移动一个棋子、主教如何移动等等。如果你有雄心壮志，可以自己实现游戏的其余部分。

总结要点 ​

  • 使用 class 关键字声明类。使用 extends 关键字扩展它们。
  • 类可以是具体的或抽象的。抽象类可以有抽象方法和抽象属性。
  • 方法可以是私有的、受保护的，或者默认情况下是公共的。它们可以是实例方法或静态方法。
  • 类可以有实例属性，这些属性也可以是私有的、受保护的或公共的。你可以在声明时将实例属性标记为只读。
  • 你可以将实例属性标记为只读。

super ​

像 JavaScript 一样，TypeScript 支持 super 调用。如果你的子类重写了父类上定义的方法（比如，如果 Queen 和 Piece 都实现了 take 方法），子类实例可以进行 super 调用来调用其父类的方法版本（例如，super.take()）。有两种 super 调用：

两种 super 调用类型 ​

  1. 方法调用，如 super.take
  2. 构造函数调用，具有特殊形式 super() 且只能从构造函数中调用

构造函数调用：如果你的子类有构造函数，你必须从子类的构造函数中调用 super() 来正确设置类（不用担心；如果你忘记了，TypeScript 会提醒你，就像一个很酷的功能机器人大象一样）。

注意，你只能使用 super 访问父类的方法，而不是其属性。

使用 this 作为返回类型 ​

就像你可以使用 this 作为值一样，你也可以将其用作类型（就像我们在"类型化 this"中所做的那样）。在处理类时，this 类型对于注释方法的返回类型很有用。

例如，让我们构建一个支持两个操作的 ES6 Set 数据结构的简化版本：向集合添加数字，以及检查给定数字是否在集合中。你可以这样使用它：

typescript
let set = new Set
set.add(1).add(2).add(3)
set.has(2) // true
set.has(4) // false

让我们定义 Set 类，从 has 方法开始：

typescript
class Set {
  has(value: number): boolean {
    // ...
  }
}

add 怎么样？当你调用 add 时，你会得到一个 Set 的实例。我们可以这样类型化：

typescript
class Set {
  has(value: number): boolean {
    // ...
  }
  add(value: number): Set {
    // ...
  }
}

到目前为止，一切都很好。当我们尝试子类化 Set 时会发生什么？

子类化的问题 ​

typescript
class MutableSet extends Set {
  delete(value: number): boolean {
    // ...
  }
}

当然，Set 的 add 方法仍然返回一个 Set，我们需要为我们的子类重写它：

typescript
class MutableSet extends Set {
  delete(value: number): boolean {
    // ...
  }
  add(value: number): MutableSet {
    // ...
  }
}

在处理扩展其他类的类时，这可能会变得有点繁琐——你必须重写每个返回 this 的方法的签名。如果你最终要重写每个方法来取悦类型检查器，那么从基类继承有什么意义呢？

使用 this 作为返回类型的解决方案 ​

相反，你可以使用 this 作为返回类型注释，让 TypeScript 为你完成工作：

typescript
class Set {
  has(value: number): boolean {
    // ...
  }
  add(value: number): this {
    // ...
  }
}

现在，你可以从 MutableSet 中删除 add 重写，因为 this 在 Set 中指向 Set 实例，而 this 在 MutableSet 中指向 MutableSet 实例：

typescript
class MutableSet extends Set {
  delete(value: number): boolean {
    // ...
  }
}

这是一个非常方便的功能，用于处理链式 API，就像我们在"构建器模式"中所做的那样。

接口 ​

当你使用类时，你经常会发现自己将它们与接口一起使用。

像类型别名一样，接口是一种为类型命名的方式，这样你就不必在内联中定义它。类型别名和接口主要是两种语法，用于同一件事（如函数表达式和函数声明），但它们有一些共同点。让我们从它们的共同点开始。考虑以下类型别名：

typescript
type Sushi = {
  calories: number
  salty: boolean
  tasty: boolean
}

很容易将其重写为接口：

typescript
interface Sushi {
  calories: number
  salty: boolean
  tasty: boolean
}

在你使用 Sushi 类型别名的任何地方，你也可以使用你的 Sushi 接口。两个声明定义形状，这些形状可以分配给彼此（实际上，它们是相同的）。

当你开始组合类型时，事情变得更有趣。让我们为寿司建模另一种食物：

typescript
type Cake = {
  calories: number
  sweet: boolean
  tasty: boolean
}

很多食物都有卡路里并且很美味——不仅仅是寿司和蛋糕。让我们将 Food 分解为自己的类型，并根据它重新定义我们的食物：

typescript
type Food = {
  calories: number
  tasty: boolean
}

type Sushi = Food & {
  salty: boolean
}

type Cake = Food & {
  sweet: boolean
}

几乎等价地，你可以用接口做到这一点：

typescript
interface Food {
  calories: number
  tasty: boolean
}

interface Sushi extends Food {
  salty: boolean
}

interface Cake extends Food {
  sweet: boolean
}

注意

接口不必扩展其他接口。实际上，接口可以扩展任何形状：对象类型、类或另一个接口。

类型和接口之间有什么区别？有三个，它们很微妙。

第一个是类型更通用，因为它们的右侧可以是任何类型，包括类型表达式（类型和可能一些类型操作符，如 & 或 |）；对于接口，右侧必须是形状。例如：

typescript
type A = number
type B = A | string

第二个区别是，当你扩展接口时，TypeScript 将确保你要扩展的接口可以分配给你的扩展。例如：

typescript
interface A {
  good(x: number): string
  bad(x: number): string
}

interface B extends A {
  good(x: string | number): string
  bad(x: string): string  // 错误 TS2430: 接口 'B' 错误地扩展了
}                         // 接口 'A'。类型 'number' 不能分配
                         // 给类型 'string'。

第三个区别是，同一作用域中的多个接口会自动合并；同一作用域中的多个类型别名会抛出编译时错误。我们称这种接口合并为声明合并：

```typescript
// 使用接口 - 这是有效的
interface User {
  name: string
}

interface User {
  age: number
}

let a: User = {
  name: 'Ashley',
  age: 30
}

// 使用类型别名 - 这会导致错误
type User = {
  name: string
}

type User = {  // 错误 TS2300: 重复标识符 'User'
  age: number
}

接口的特点总结 ​

  • 接口扩展：使用 extends 关键字继承其他接口
  • 类型安全：扩展时 TypeScript 会检查类型兼容性
  • 声明合并：同名接口会自动合并属性
  • 形状约束：接口只能定义对象形状，不能是联合类型等复杂类型表达式

接口为面向对象编程提供了强大的类型约束机制，特别适合定义类的契约和 API 规范。

声明合并 ​

这不是使用交集类型时的情况：如果你将最后一个示例中的接口转换为类型别名，TypeScript 将尽力将你的扩展与你要扩展的类型结合起来，从而导致重载签名错误（如果在你的代码编辑器中尝试）。

当你为对象类型建模继承时，TypeScript 对接口进行的可分配性检查可能是一个有用的工具来捕获错误。

第三个区别是，同一作用域中的多个接口会自动合并；同一作用域中的多个类型别名会抛出编译时错误。这被称为声明合并。

声明合并示例 ​

声明合并是 TypeScript 自动组合多个同名声明的方式。当使用其他功能（如命名空间声明，参见"命名空间"）时，它也会出现，在使用接口时，我们将简要介绍声明合并的上下文。有关更深入的了解，请参阅"声明合并"。

例如，如果你声明两个同名的 User 接口，那么 TypeScript 将自动为你组合它们成一个接口：

typescript
// User 有一个字段，name
interface User {
  name: string
}

// User 现在有两个字段，name 和 age
interface User {
  age: number
}

let a: User = {
  name: 'Ashley',
  age: 30
}

如果你用类型别名重复该示例，会发生什么：

typescript
type User = {  // 错误 TS2300: 重复标识符 'User'
  name: string
}

type User = {  // 错误 TS2300: 重复标识符 'User'
  age: number
}

注意，两个接口不能冲突；如果一个类型将属性声明为 T，另一个类型将其声明为 U，并且 T 和 U 不相同，那么你会得到一个错误：

typescript
interface User {
  age: string
}

interface User {
  age: number  // 错误 TS2717: 后续属性声明必须具有
}              // 相同的类型。属性 'age' 的类型必须是 'string'，
               // 但这里的类型是 'number'。

如果你的接口声明泛型（跳到"多态性"了解更多），这些泛型必须以完全相同的方式声明为两个接口；否则你会得到一个错误。

实现 ​

当你声明一个类时，你可以使用 implements 关键字来说明它满足特定的接口。像其他显式类型注释一样，这是一种方便的方式来添加类型级约束，使你的类尽可能接近实现本身，这样来自错误实现的错误就不会在下游显示，而是在它被抛出的地方显示。这是一种熟悉的实现常见设计模式的方式，如适配器、工厂和策略（参见本章末尾的一些示例）。

这就是它的样子：

typescript
interface Animal {
  eat(food: string): void
  sleep(hours: number): void
}

class Cat implements Animal {
  eat(food: string) {
    console.info('Ate some', food, '. Meow!')
  }
  sleep(hours: number) {
    console.info('Slept for', hours, 'hours')
  }
}

有趣的是，这是 TypeScript 检查两种类型不仅可分配，而且相同的罕见地方之一。

接口的关键特性总结 ​

1. 声明合并的优势 ​

  • 模块化定义：可以在不同地方扩展同一个接口
  • 库扩展：允许为第三方库添加类型定义
  • 渐进式开发：支持逐步完善接口定义

2. 接口实现的好处 ​

  • 契约约束：确保类实现所有必需的方法
  • 设计模式支持：便于实现适配器、工厂、策略等模式
  • 类型安全：在编译时捕获实现错误
  • 文档作用：接口作为类的使用说明

3. 接口 vs 类型别名的选择指南 ​

使用接口当：

  • 定义对象的形状
  • 需要继承或实现
  • 可能需要声明合并
  • 定义类的契约

使用类型别名当：

  • 定义联合类型、交集类型
  • 创建复杂的类型表达式
  • 需要类型计算
  • 定义函数类型

接口是 TypeScript 面向对象编程的核心工具，提供了强大的类型约束和代码组织能力。

接口实现的详细规则 ​

Cat 必须实现 Animal 声明的每个方法，并且可以在此基础上实现更多方法和属性，如果它想要的话。

接口可以声明实例属性，但它们不能声明可见性修饰符（或 private、protected 和 public）并且它们不能使用 static 关键字。你可以将实例属性标记为 readonly，就像我们对对象类型在对象中所做的那样（在第3章中）：

typescript
interface Animal {
  readonly name: string
  eat(food: string): void
  sleep(hours: number): void
}

你不仅限于实现一个接口——你可以实现任意多个你想要的接口：

typescript
interface Animal {
  readonly name: string
  eat(food: string): void
  sleep(hours: number): void
}

interface Feline {
  meow(): void
}

class Cat implements Animal, Feline {
  name = 'Whiskers'
  eat(food: string) {
    console.info('Ate some', food, '. Mmm!')
  }
  sleep(hours: number) {
    console.info('Slept for', hours, 'hours')
  }
  meow() {
    console.info('Meow')
  }
}

所有这些功能都是完全类型安全的。如果你忘记实现一个方法或属性，或者实现不正确，TypeScript 将前来救援（参见图5-3）。

实现接口 vs 扩展抽象类 ​

实现接口实际上类似于扩展抽象类。区别在于接口更通用和轻量级，而抽象类更专用和功能丰富。

接口是一种建模形状的方式。在值级别，这意味着对象、数组、函数、类或类实例。接口不会发出 JavaScript 代码，只存在于编译时。

抽象类只能建模，嗯，一个类。它发出运行时代码，即你猜对了，一个 JavaScript 类。抽象类可以有构造函数，提供默认实现，并为属性和方法设置访问修饰符。接口不能做任何这些事情。

你使用哪一个取决于你的用例。当实现在多个类之间共享时，使用抽象类。当你需要一种轻量级的方式来说"这个类是 T"时，使用接口。

选择指南 ​

使用接口当：

  • 需要定义多个不相关类的共同契约
  • 要求轻量级的类型约束
  • 需要支持多重实现
  • 只关心类型形状，不关心实现

使用抽象类当：

  • 多个类需要共享代码实现
  • 需要构造函数逻辑
  • 要使用访问修饰符
  • 需要在运行时检查类型

类的高级特性 ​

类的结构化类型 ​

TypeScript 对类使用结构化类型系统。这意味着如果两个类具有相同的形状，TypeScript 会认为它们是兼容的，即使它们没有明确的继承关系：

typescript
class Zebra {
  trot() {
    // ...
  }
}

class Poodle {
  trot() {
    // ...
  }
}

function ambleAround(animal: Zebra) {
  animal.trot()
}

let zebra = new Zebra
let poodle = new Poodle

ambleAround(zebra)   // OK
ambleAround(poodle)  // OK，因为 Poodle 具有 Zebra 的形状

类的类型注解 ​

类既可以作为值使用，也可以作为类型使用：

typescript
class Cat {
  name: string
  constructor(name: string) {
    this.name = name
  }
}

// Cat 作为类型使用
let cat: Cat = new Cat('Fluffy')

// Cat 作为值使用
let CatConstructor = Cat
let anotherCat = new CatConstructor('Mittens')

错误处理示例 ​

当类没有正确实现接口时，TypeScript 会提供清晰的错误信息：

typescript
interface Feline {
  meow(): void
}

class Cat implements Feline {
  name = 'Whiskers'
  // 错误：类 'Cat' 错误地实现了接口 'Feline'
  // 类型 'Cat' 中缺少属性 'meow'，但在类型 'Feline' 中是必需的
}

这种类型检查确保了代码的可靠性和一致性。

类的结构化类型 ​

与 TypeScript 中的其他类型一样，TypeScript 通过结构而不是名称来比较类。如果一个类与另一个类型兼容，那么它与定义相同属性或方法的任何其他对象都兼容。这对于来自 Java、Scala 和大多数其他语言的人来说很重要，在这些语言中，类通常是名义类型的。这意味着如果你有一个接受 Zebra 的函数，并且你给它一个 Poodle，TypeScript 可能不会介意！

typescript
class Zebra {
  trot() {
    // ...
  }
}

class Poodle {
  trot() {
    // ...
  }
}

function ambleAround(animal: Zebra) {
  animal.trot()
}

let zebra = new Zebra
let poodle = new Poodle

ambleAround(zebra)   // OK
ambleAround(poodle)  // OK

正如你所知道的系统发生学家，斑马不是贵宾犬——但 TypeScript 不介意！只要 Poodle 可以分配给 Zebra，TypeScript 就可以接受，因为从我们函数的角度来看，我们使用的是实现 trot() 的东西。如果你使用几乎任何其他类型类别的语言，这段代码会引发错误，但 TypeScript 通过结构类型是完全可以接受的。

私有和受保护字段的特殊规则 ​

此规则的例外是具有 private 或 protected 字段的类：在检查形状是否可分配给类时，如果类具有任何 private 或 protected 字段并且形状不是该类或该类的子类的实例，则形状不可分配给类：

typescript
class A {
  private x = 1
}

class B extends A {}

function f(a: A) {}

f(new A)  // OK
f(new B)  // OK
f({x: 1}) // 错误 TS2345: 类型 '{x: number}' 的参数不能
          // 分配给类型 'A' 的参数。属性 'x' 是私有的
          // 在类型 'A' 中，但在类型 '{x: number}' 中不是。

类声明值和类型 ​

你可以在 TypeScript 中表达的大多数东西要么是值要么是类型：

typescript
// 值
let a = 1999
function b() {}

// 类型
type a = number
interface b {
  (): void
}

类型和值在 TypeScript 中是分别命名空间的。根据你如何使用术语（在这个例子中是 a 或 b），TypeScript 会推断你是指值还是类型：

typescript
if (a + 1 > 3) // ... // TypeScript 从上下文推断你指的是值 a
let x: a = 3          // TypeScript 从上下文推断你指的是类型 a

这种上下文术语解析真的很好，让我们做一些很酷的事情，比如实现伴生对象类型（参见"伴生对象模式"）。

类和枚举是特殊的。它们是唯一的，因为它们在类型命名空间和值命名空间中都生成一个类型：

typescript
class C {}
let c: C = new C()

enum E {Red}
let e: E = E.Red

在这个上下文中，C 指的是我们 C 类的实例类型。 在这个上下文中，C 指的是 C 的值。 在这个上下文中，E 指的是我们 E 枚举的类型。 在这个上下文中，E 指的是 E 的值。

类的双重性质示例 ​

这种双重性质使得类在 TypeScript 中非常灵活：

typescript
class StringDatabase {
  state: string[] = []

  get(index: number): string | undefined {
    return this.state[index]
  }

  set(index: number, value: string): void {
    this.state[index] = value
  }
}

// StringDatabase 作为类型使用
let db: StringDatabase = new StringDatabase()

// StringDatabase 作为值使用（构造函数）
let DatabaseConstructor = StringDatabase
let anotherDb = new DatabaseConstructor()

// 可以将类作为参数传递
function createDatabase(DatabaseClass: typeof StringDatabase) {
  return new DatabaseClass()
}

let thirdDb = createDatabase(StringDatabase)

结构化类型的实际应用 ​

结构化类型在实际开发中非常有用，特别是在处理第三方库或不同模块之间的交互时：

typescript
// 来自不同模块的类
class APIUser {
  name: string
  email: string

  constructor(name: string, email: string) {
    this.name = name
    this.email = email
  }

  sendEmail() {
    console.log(`Sending email to ${this.email}`)
  }
}

class DatabaseUser {
  name: string
  email: string

  constructor(name: string, email: string) {
    this.name = name
    this.email = email
  }

  sendEmail() {
    console.log(`Sending email to ${this.email}`)
  }
}

// 这个函数可以接受任何具有相同结构的对象
function notifyUser(user: APIUser) {
  user.sendEmail()
}

let apiUser = new APIUser('Alice', 'alice@example.com')
let dbUser = new DatabaseUser('Bob', 'bob@example.com')

notifyUser(apiUser)  // OK
notifyUser(dbUser)   // OK - 结构兼容
notifyUser({         // OK - 对象字面量也兼容
  name: 'Charlie',
  email: 'charlie@example.com',
  sendEmail() {
    console.log(`Sending email to ${this.email}`)
  }
})

这种灵活性使得 TypeScript 既保持了类型安全，又不会过于严格地限制代码的可重用性。

类的实例化和构造函数 ​

当我们使用类时，我们需要一种方式来表达"这个变量应该是这个类的实例"，以及"这个变量应该是这个类"。因为类和枚举在类型级别生成类型，我们能够表达这种"是一个"关系。

我们还需要一种在运行时表示类的方式，以便我们可以用 new 实例化它，调用其静态方法，进行元编程，并使用 instanceof 操作它——所以类需要生成一个值。

在前面的例子中，C 指的是类 C 的实例。我们如何谈论 C 类本身？我们使用 typeof 关键字（TypeScript 提供的类型操作符，类似于 JavaScript 的值级 typeof，但用于类型）。

让我们创建一个类 StringDatabase——世界上最简单的数据库：

typescript
type State = {
  [key: string]: string
}

class StringDatabase {
  state: State = {}

  get(key: string): string | null {
    return key in this.state ? this.state[key] : null
  }

  set(key: string, value: string): void {
    this.state[key] = value
  }

  static from(state: State) {
    let db = new StringDatabase
    for (let key in state) {
      db.set(key, state[key])
    }
    return db
  }
}

这个类声明生成了什么类型？实例类型 StringDatabase：

typescript
interface StringDatabase {
  state: State
  get(key: string): string | null
  set(key: string, value: string): void
}

以及构造函数类型 typeof StringDatabase：

typescript
interface StringDatabaseConstructor {
  new(): StringDatabase
  from(state: State): StringDatabase
}

构造函数签名 ​

那个 new() 被称为构造函数签名，它是 TypeScript 表达给定类型可以用 new 实例化的方式。因为 TypeScript 是结构类型的，我们能够做的最酷的事情就是描述什么是类：类是任何可以被 new 的东西。

在这种情况下，构造函数不接受任何参数，但你可以使用它来声明构造函数：

typescript
class StringDatabase {
  constructor(public state: State = {}) {}
  // ...
}

我们可以这样类型化 StringDatabase 的构造函数签名：

typescript
interface StringDatabaseConstructor {
  new(state?: State): StringDatabase
  from(state: State): StringDatabase
}

所以，类声明不仅在值和类型级别生成术语，而且在类型级别生成两个术语：一个表示类的实例；一个表示类构造函数本身（可通过 typeof 类型操作符访问）。

多态性 ​

像函数和类型别名一样，类和接口对泛型类型参数有丰富的支持，包括默认值和边界。你可以将泛型作用域限定到整个类或接口，或者限定到特定方法：

typescript
class MyMap<K, V> {
  constructor(initialKey: K, initialValue: V) {
    // ...
  }

  get(key: K): V {
    // ...
  }

  set(key: K, value: V): void {
    // ...
  }

  merge<K1, V1>(map: MyMap<K1, V1>): MyMap<K | K1, V | V1> {
    // ...
  }
}

泛型类的使用示例 ​

typescript
// 创建一个字符串到数字的映射
let stringToNumber = new MyMap<string, number>('initial', 42)

// 创建一个数字到布尔值的映射
let numberToBoolean = new MyMap<number, boolean>(1, true)

// 合并两个不同类型的映射
let merged = stringToNumber.merge(numberToBoolean)
// merged 的类型是 MyMap<string | number, number | boolean>

泛型的绑定规则 ​

在使用泛型时，需要了解以下绑定规则：

  1. 类级泛型：当你声明类时绑定类级泛型类型。这里，K 和 V 对 MyMap 的每个实例方法和实例属性都可用。

  2. 方法级泛型：注意你也可以在类内部声明泛型类型。相反，将声明移到你的类声明中。

  3. 类内部使用：在类内部的任何地方使用类级泛型。

  4. 实例方法访问：实例方法可以访问类级泛型，也可以声明自己的泛型。例如，merge 使用了类级泛型 K 和 V，同时也声明了自己的泛型 K1 和 V1。

  5. 静态方法限制：静态方法无法访问其类的泛型，就像在值级别它们无法访问 K 和 V 一样。相反，它声明自己的 K 和 V 泛型。

你也可以将泛型绑定到接口：

typescript
interface MyMap<K, V> {
  get(key: K): V
  set(key: K, value: V): void
}

像函数一样，你可以将具体类型绑定到泛型，或者让 TypeScript 为你推断类型：

typescript
let a = new MyMap<string, number>('x', 1) // MyMap<string, number>
let b = new MyMap('x', true) // MyMap<string, boolean>

a.get('x') // number
b.set('x', false)

Mixins（混入） ​

JavaScript 和 TypeScript 没有 trait 或 mixin 关键字，但很容易自己实现它们。两者都是模拟多重继承的方式（类扩展多个其他类）并进行基于角色的编程，这是一种编程风格，你描述事物的"能力"而不是"是什么"关系，你描述"能够"关系。

让我们构建一个 mixin 实现。

Mixins 的特点 ​

Mixins 是一种允许我们将行为和属性混入到类中的模式。按照惯例，mixins：

  • 可以有状态（即实例属性）
  • 只能提供具体方法（不是抽象方法）
  • 可以有构造函数，这些构造函数按照它们的类被混入的相同顺序调用

TypeScript 没有内置的 mixins 概念，但很容易实现它们。例如，让我们创建一个调试库。我们将称之为 EZDebug。该库的工作原理是让你记录关于任何类使用的信息，这样你就可以在运行时检查它们，就像这样：

typescript
class User {
  // ...
}

User.debug() // 评估为 'User({"id": 3, "name": "Emma Gluzman"})'

使用标准的 .debug 接口，我们的用户将能够调试任何东西！让我们构建它。我们将使用 mixin 模式，它是一个函数，接受一个构造函数并返回一个类构造函数，所以我们的 mixin 可能看起来像这样：

typescript
type ClassConstructor = new(...args: any[]) => {}

function withEZDebug<C extends ClassConstructor>(Class: C) {
  return class extends Class {
    debug() {
      let Name = this.constructor.name
      let value = this.constructor.name
      for (let key in this) {
        value += `${JSON.stringify(key)}: ${this[key]}\n`
      }
      return `${Name}(${JSON.stringify(value)})`
    }
  }
}

使用 Mixins ​

我们首先声明一个类型 ClassConstructor，它表示任何构造函数。由于 TypeScript 是结构类型的，我们不知道构造函数可能有什么类型的参数，所以我们说它接受任意数量的任何类型的参数。

我们声明我们的 withEZDebug mixin 使用单一类型参数 C。C 必须至少是一个类构造函数，我们使用 extends 子句强制执行。我们让 TypeScript 推断 withEZDebug 的返回类型，这是 C 和我们新匿名类的交集。

由于 mixin 是一个函数，它接受一个构造函数并返回一个构造函数，所以我们返回一个匿名类构造函数。

Mixin 的实现细节 ​

类构造函数必须接受至少类你传入的类可能接受的参数。但请记住，由于我们不知道你可能传入什么类，我必须保持它尽可能通用，这意味着任何数量的任何类型的参数——就像 ClassConstructor。

最后，由于这个匿名类扩展另一个类，为了正确连接一切，我们需要记住调用 Class 的构造函数。

像常规 TypeScript 类一样，如果你的构造函数中没有更多逻辑，你可以省略 ● 和 ●。我们不会在构造函数中放入任何逻辑，所以我们可以省略它。

现在我们已经设置了样板，是时候进行一些调试魔法了。当我们调用 .debug 时，我们想要记录类的构造函数名称和实例的值：

typescript
type ClassConstructor<T = {}> = new(...args: any[]) => T

function withEZDebug<C extends ClassConstructor>(Class: C) {
  return class extends Class {
    debug() {
      let Name = this.constructor.name
      let value = this.constructor.name
      for (let key in this) {
        value += `${JSON.stringify(key)}: ${this[key]}\n`
      }
      return `${Name}(${JSON.stringify(value)})`
    }
  }
}

但等等！我们如何确保类实现了 .getDebugValue 方法，以便我们可以调用它？想想这个一秒钟，然后再继续——你能想出来吗？

答案是，我们不是接受任何旧类，而是使用泛型约束，确保传递给 withEZDebug 的类至少定义了 .getDebugValue 方法：

typescript
type ClassConstructor<T = {}> = new(...args: any[]) => T

function withEZDebug<C extends ClassConstructor<{
  getDebugValue(): object
}>>(Class: C) {
  return class extends Class {
    debug() {
      let Name = this.constructor.name
      let value = this.getDebugValue()
      return `${Name}(${JSON.stringify(value)})`
    }
  }
}

我们为 ClassConstructor 添加了一个泛型类型参数。我们将形状类型绑定到 ClassConstructor，C，强制传递给 withEZDebug 的构造函数至少定义 .getDebugValue 方法。

就是这样！现在，你如何使用这个令人难以置信的调试实用程序？像这样：

typescript
class HardToDebugUser {
  constructor(
    private id: number,
    private firstName: string,
    private lastName: string
  ) {}
  getDebugValue() {
    return {
      id: this.id,
      name: this.firstName + ' ' + this.lastName
    }
  }
}

let User = withEZDebug(HardToDebugUser)
let user = new User(3, 'Emma', 'Gluzman')
user.debug() // 评估为 'User({"id": 3, "name": "Emma Gluzman"})'

很酷，对吧？你可以将尽可能多的 mixins 应用到一个类，以产生具有更丰富和更丰富行为的类，并且是表达特定可重用行为的一种方式。

Decorators（装饰器） ​

装饰器是一个实验性的 TypeScript 功能，它为我们提供了一个干净的语法来进行元编程，使用类方法、类属性和方法参数。它们只是调用你正在装饰的东西的函数的语法糖。

TSC FLAG: EXPERIMENTALDECORATORS ​

因为它们仍然是实验性的——意味着它们可能在向后不兼容的方式中改变，或者可能在未来的 TypeScript 版本中完全删除——装饰器隐藏在 TSC 标志后面。如果你对此功能感兴趣，并想要使用该功能，请在你的 tsconfig.json 中设置 "experimentalDecorators": true，然后阅读。

要了解装饰器的工作原理，让我们从一个例子开始：

typescript
@serializable
class APIPayload {
  getValue(): Payload {
    // ...
  }
}

@serializable 类装饰器包装我们的 APIPayload 类，并可选地返回一个替换它的新类。没有装饰器，你可能会实现同样的事情：

typescript
let APIPayload = serializable(class APIPayload {
  getValue(): Payload {
    // ...
  }
})

对于每种类型的装饰器，TypeScript 要求你有一个具有给定名称的函数在作用域内，以及给定类型的装饰器的正确类型签名。

表 5-1. 不同类型装饰器函数的预期类型签名 ​

你正在装饰的内容预期类型签名
类(constructor: {new(...args: any[]): any}) => any
方法(classPrototype: {}, methodName: string, descriptor: PropertyDescriptor) => any
静态方法(constructor: {new(...args: any[]): any}, methodName: string, descriptor: PropertyDescriptor) => any
方法参数(classPrototype: {}, methodName: string, index: number) => void
静态方法参数(constructor: {new(...args: any[]): any}, paramName: string, index: number) => void
属性(classPrototype: {}, propertyName: string) => any
静态属性(constructor: {new(...args: any[]): any}, propertyName: string) => any
属性获取器/设置器(classPrototype: {}, propertyName: string, descriptor: PropertyDescriptor) => any
静态属性获取器/设置器(constructor: {new(...args: any[]): any}, propertyName: string, descriptor: PropertyDescriptor) => any

记住，new() 是我们在 TypeScript 中结构化类型化类构造函数的方式。对于类构造函数，它扩展了 ClassConstructor，我们传递了一个具有任何扩展的 spread: new(...any[])。

@serializable 可以装饰任何其实例实现方法 .getValue 的类，它返回一个 Payload：

typescript
let payload = new APIPayload
let serialized = payload.serialize() // 错误 TS2339: 属性 'serialize' 不存在于类型 'APIPayload' 上

TypeScript 假设装饰器不会改变它正在装饰的东西的形状——意味着你不能添加或删除方法和属性。它在编译时检查类你返回的是可分配给你传入的类，在运行时你的装饰器。TypeScript 不会跟踪你在装饰器中所做的扩展。

直到装饰器在 TypeScript 中变得更加成熟，我建议你避免使用它们并坚持常规函数：

typescript
let DecoratedAPIPayload = serializable(APIPayload)
let payload = new DecoratedAPIPayload
let serialized = payload.serialize() // 错误 TS2339: 属性 'serialize' 不存在于类型 'APIPayload' 上

我们不会在本书中更深入地研究装饰器。有关更多信息，请前往官方文档。

模拟 final 类 ​

虽然 TypeScript 不支持类或方法的 final 关键字，但如果你以前没有使用过面向对象语言，final 是关键字，一些语言使用它来标记类为不可扩展或方法为不可重写。

要在 TypeScript 中模拟 final 类，我们可以利用私有构造函数：

typescript
class MessageQueue {
  private constructor(private messages: string[]) {}
}

当构造函数标记为 private 时，你不能 new 该类或扩展它：

typescript
class BadQueue extends MessageQueue {} // 错误 TS2675: 无法扩展类 'MessageQueue'。类构造函数标记为私有

new MessageQueue([]) // 错误 TS2673: 类 'MessageQueue' 的构造函数是私有的，只能在类声明内访问

以及防止你扩展类——这就是我们想要的——私有构造函数也防止你直接实例化它。但对于 final 类，我们确实希望能够实例化它，只是不扩展它。我们如何保持第一个限制但摆脱第二个？简单：

typescript
class MessageQueue {
  private constructor(private messages: string[]) {}
  static create(messages: string[]) {
    return new MessageQueue(messages)
  }
}

这改变了 MessageQueue 的 API 一点，但它很好地防止了编译时的扩展：

typescript
class BadQueue extends MessageQueue {} // 错误 TS2675: 无法扩展类 'MessageQueue'。类构造函数标记为私有

MessageQueue.create([]) // MessageQueue

TypeScript 没有内置装饰器：无论你使用什么装饰器，你都必须自己实现。实现取决于你正在装饰的内容——对于类、方法、属性和函数参数，这是一个满足特定签名的常规函数。例如，我们的 @serializable 装饰器可能看起来像这样：

typescript
type ClassConstructor<T = {}> = new(...args: any[]) => T

function serializable<
  T extends ClassConstructor<{
    getValue(): Payload
  }>
>(constructor: T) {
  return class extends constructor {
    serialize() {
      return this.getValue().toString()
    }
  }
}

这不会是关于面向对象编程的章节，如果我们不通过实现设计模式或两个来走过 TypeScript，对吧？

设计模式 ​

Factory Pattern（工厂模式） ​

factory pattern 是一种创建某种类型对象的方式，将决定创建哪个具体对象的决定留给创建该对象的特定工厂。

让我们构建一个鞋子工厂。我们将首先定义一个 Shoe 类型，以及两种鞋子：

typescript
type Shoe = {
  purpose: string
}

class Ballerina implements Shoe {
  purpose = 'dancing'
}

class Boot implements Shoe {
  purpose = 'woodcutting'
}

class Sneaker implements Shoe {
  purpose = 'walking'
}

注意这个例子使用了一个 type，但我们也可以使用一个 interface。

现在，让我们制作一个鞋子工厂：

typescript
let Shoe = {
  create(type: 'ballerina' | 'boot' | 'sneaker'): Shoe {
    switch (type) {
      case 'ballerina': return new Ballerina
      case 'boot': return new Boot
      case 'sneaker': return new Sneaker
    }
  }
}
  • 使用联合类型作为 type 有助于使 .create 尽可能类型安全，防止消费者传递无效的鞋子类型。
  • 在 type 上切换使 TypeScript 容易强制我们已经处理了 Shoe 的每种类型。

在这个例子中，我们使用伴随对象模式（参见"伴随对象模式"）来声明一个类型 Shoe 和一个值 Shoe，具有相同的名称（记住 TypeScript 为值和类型有单独的命名空间），作为向值提供方法的一种方式来操作类型。要使用工厂，我们可以调用 .create：

typescript
Shoe.create('ballerina') // Ballerina

太棒了！我们有一个工厂模式。注意我们可以进一步指示 Shoe.create 的类型签名，传入 'boot' 将给出 Boot，'sneaker' 将给出 Sneaker，等等，但这会破坏工厂模式给我们的抽象（消费者不应该知道他们得到什么类，只是该类满足特定接口）。

Builder Pattern（建造者模式） ​

builder pattern 是一种将对象的构造与对象实际实现的方式分离的方法。如果你使用过 jQuery 或 ES6 数据结构如 Map 和 Set，这种 API 风格应该看起来很熟悉。这是它的样子：

typescript
new RequestBuilder()
  .setURL('/users')
  .setMethod('get')
  .setData({firstName: 'Anna'})
  .send()

我们如何实现 RequestBuilder？简单——我们将从一个基本类开始：

typescript
class RequestBuilder {}

首先我们将添加 .setURL 方法：

typescript
class RequestBuilder {
  private url: string | null = null ①

  setURL(url: string): this { ②
    this.url = url
    return this
  }
}

① 我们在私有实例变量 url 中跟踪用户设置的 URL，我们将其初始化为 null。 ② setURL 的返回类型是 this（参见"使用 this 作为返回类型"），即用户调用 setURL 的 RequestBuilder 的特定实例。

现在让我们从示例中添加其他方法：

typescript
class RequestBuilder {
  private data: object | null = null
  private method: 'get' | 'post' | null = null
  private url: string | null = null

  setMethod(method: 'get' | 'post'): this {
    this.method = method
    return this
  }

  setData(data: object): this {
    this.data = data
    return this
  }

  setURL(url: string): this {
    this.url = url
    return this
  }

  send() {
    // ...
  }
}

就是这样。

注意

这种传统的建造者设计并不完全安全：我们可以调用 .send 在我们设置方法、URL 或数据之前，导致运行时异常（记住，这是坏的异常类型）。参见练习 4 了解一些关于如何改进这种设计的想法。

练习 ​

1. 类和接口的区别 ​

问题：类和接口之间有什么区别？

答案：

类和接口的主要区别包括：

类（Class）：

  • 既是类型又是值
  • 可以有实现（方法体、属性初始值）
  • 可以有构造函数
  • 可以被实例化
  • 支持继承（extends）
  • 可以有访问修饰符（private、protected、public）
  • 可以有静态成员
  • 在运行时存在

接口（Interface）：

  • 只是类型，不是值
  • 只能定义形状，不能有实现
  • 不能有构造函数
  • 不能被实例化
  • 支持扩展（extends）和实现（implements）
  • 所有成员都是公共的
  • 不能有静态成员
  • 在运行时不存在（编译后被擦除）
typescript
// 接口示例
interface Animal {
  name: string
  makeSound(): void
}

// 类示例
class Dog implements Animal {
  constructor(public name: string) {}

  makeSound() {
    console.log('Woof!')
  }
}

// 类既是类型又是值
let dogType: Dog = new Dog('Buddy') // Dog 作为类型和构造函数

2. protected 构造函数 ​

问题：当你将类的构造函数标记为 private 时，意味着你不能实例化或扩展该类。当你将其标记为 protected 时会发生什么？在你的代码编辑器中尝试一下，看看你能否弄清楚。

答案：

当构造函数标记为 protected 时：

  • 不能直接实例化该类
  • 可以被子类扩展
  • 子类可以调用父类的 protected 构造函数
typescript
class BaseClass {
  protected constructor(protected value: string) {}
}

// 错误：不能实例化
// let base = new BaseClass('test') // Error: Constructor of class 'BaseClass' is protected

// 正确：可以扩展
class DerivedClass extends BaseClass {
  constructor(value: string, public extra: number) {
    super(value) // 可以调用父类的 protected 构造函数
  }
}

// 可以实例化子类
let derived = new DerivedClass('test', 42) // OK

这种模式常用于创建抽象基类，强制通过子类来使用功能。

3. 改进工厂模式 ​

问题：扩展我们开发的"工厂模式"实现，使其更安全，但代价是稍微破坏抽象。更新实现，使消费者在编译时知道调用 Shoe.create('boot') 返回 Boot，调用 Shoe.create('balletFlat') 返回 BalletFlat（而不是两者都返回 Shoe）。提示：回想"重载函数类型"。

答案：

使用函数重载来提供更精确的类型信息：

typescript
// 鞋子类型
type ShoeType = 'boot' | 'balletFlat' | 'sneaker'

// 基础鞋子类
abstract class Shoe {
  abstract protect(): void

  // 函数重载声明
  static create(type: 'boot'): Boot
  static create(type: 'balletFlat'): BalletFlat
  static create(type: 'sneaker'): Sneaker
  static create(type: ShoeType): Shoe

  // 实际实现
  static create(type: ShoeType): Shoe {
    switch (type) {
      case 'boot':
        return new Boot()
      case 'balletFlat':
        return new BalletFlat()
      case 'sneaker':
        return new Sneaker()
      default:
        throw new Error(`Unknown shoe type: ${type}`)
    }
  }
}

class Boot extends Shoe {
  protect() {
    return 'boots'
  }
}

class BalletFlat extends Shoe {
  protect() {
    return 'ballet flats'
  }
}

class Sneaker extends Shoe {
  protect() {
    return 'sneakers'
  }
}

// 现在类型是精确的
let boot = Shoe.create('boot')           // 类型：Boot
let flat = Shoe.create('balletFlat')     // 类型：BalletFlat
let sneaker = Shoe.create('sneaker')     // 类型：Sneaker

这种方法提供了类型安全性，同时保持了工厂模式的基本结构。

4. 类型安全的建造者模式 ​

问题：[困难] 作为练习，思考如何设计一个类型安全的建造者模式。扩展建造者模式"Builder Pattern"以：

a. 在编译时保证某人不能在至少设置 URL 和方法之前调用 .send。如果你还强制用户按特定顺序调用方法，会更容易做出这种保证吗？（提示：你可以返回什么而不是 this？）

b. [更困难] 如果你想做出这种保证，但仍然让人们以任何顺序调用方法，你会如何改变你的设计？（提示：TypeScript 的什么功能可以让你在每次方法调用后将每个方法的返回类型"添加"到 this 类型？）

答案：

解决方案 a：强制特定顺序 ​

typescript
// 使用状态类型来强制顺序
class RequestBuilder {
  private constructor() {}

  static create(): RequestBuilderWithoutURL {
    return new RequestBuilderWithoutURL()
  }
}

class RequestBuilderWithoutURL {
  url(url: string): RequestBuilderWithURL {
    return new RequestBuilderWithURL(url)
  }
}

class RequestBuilderWithURL {
  constructor(private _url: string) {}

  method(method: 'GET' | 'POST' | 'PUT' | 'DELETE'): RequestBuilderComplete {
    return new RequestBuilderComplete(this._url, method)
  }
}

class RequestBuilderComplete {
  constructor(
    private _url: string,
    private _method: string,
    private _data?: any,
    private _headers?: Record<string, string>
  ) {}

  data(data: any): RequestBuilderComplete {
    return new RequestBuilderComplete(this._url, this._method, data, this._headers)
  }

  headers(headers: Record<string, string>): RequestBuilderComplete {
    return new RequestBuilderComplete(this._url, this._method, this._data, headers)
  }

  send(): Promise<Response> {
    // 实际发送请求的逻辑
    return fetch(this._url, {
      method: this._method,
      body: this._data ? JSON.stringify(this._data) : undefined,
      headers: this._headers
    })
  }
}

// 使用示例
let request = RequestBuilder
  .create()
  .url('https://api.example.com/users')
  .method('POST')
  .data({ name: 'John' })
  .headers({ 'Content-Type': 'application/json' })
  .send() // 只有在设置了 URL 和 method 后才能调用

// 错误示例：
// RequestBuilder.create().send() // Error: send 方法不存在

解决方案 b：任意顺序调用（使用交集类型） ​

typescript
// 使用交集类型和条件类型
interface HasURL {
  _hasURL: true
}

interface HasMethod {
  _hasMethod: true
}

interface RequestConfig {
  url?: string
  method?: 'GET' | 'POST' | 'PUT' | 'DELETE'
  data?: any
  headers?: Record<string, string>
}

class FlexibleRequestBuilder<T = {}> {
  constructor(private config: RequestConfig = {}) {}

  url(url: string): FlexibleRequestBuilder<T & HasURL> {
    return new FlexibleRequestBuilder({
      ...this.config,
      url
    }) as any
  }

  method(method: 'GET' | 'POST' | 'PUT' | 'DELETE'): FlexibleRequestBuilder<T & HasMethod> {
    return new FlexibleRequestBuilder({
      ...this.config,
      method
    }) as any
  }

  data(data: any): FlexibleRequestBuilder<T> {
    return new FlexibleRequestBuilder({
      ...this.config,
      data
    }) as any
  }

  headers(headers: Record<string, string>): FlexibleRequestBuilder<T> {
    return new FlexibleRequestBuilder({
      ...this.config,
      headers
    }) as any
  }

  // 只有当 T 包含 HasURL 和 HasMethod 时才能调用 send
  send(this: FlexibleRequestBuilder<HasURL & HasMethod>): Promise<Response> {
    return fetch(this.config.url!, {
      method: this.config.method!,
      body: this.config.data ? JSON.stringify(this.config.data) : undefined,
      headers: this.config.headers
    })
  }
}

// 使用示例 - 任意顺序
let builder = new FlexibleRequestBuilder()

// 可以以任意顺序调用
let request1 = builder
  .method('POST')
  .url('https://api.example.com/users')
  .data({ name: 'John' })
  .send() // OK

let request2 = builder
  .url('https://api.example.com/users')
  .headers({ 'Authorization': 'Bearer token' })
  .method('GET')
  .send() // OK

// 错误示例：
// builder.url('test').send() // Error: 缺少 method
// builder.method('GET').send() // Error: 缺少 url

这两种解决方案展示了 TypeScript 类型系统的强大功能：

  • 方案 a 使用不同的类来强制执行状态转换
  • 方案 b 使用泛型和交集类型来跟踪已设置的属性，允许更灵活的 API 使用
