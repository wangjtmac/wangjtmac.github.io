第7章 处理错误 ​

章节引言 ​

这章用了一个生动的故事开头：三个专业人士（物理学家、结构工程师、程序员）的车子在山路上刹车失灵了。面对危险：

  • 物理学家想分析刹车片的摩擦力和温度
  • 结构工程师想拿后备箱的扳手工具检查车辆问题
  • 程序员说："我们先看看这个问题能否重现？"

核心概念 ​

TypeScript 的错误处理优势：

  • TypeScript 努力把运行时异常转移到编译时
  • 丰富的类型系统 + 静态分析 = 减少运行时错误
  • 不用再熬夜调试拼写错误的变量名和空指针异常

现实情况： 虽然 TypeScript 很强大，但有些运行时错误还是防不住：

  • 网络故障
  • 文件系统错误
  • 用户输入解析错误
  • 内存溢出等

本章要学的四种错误处理方式 ​

  1. 返回 null - 最简单直接
  2. 抛出异常 - 传统做法
  3. 返回异常 - 函数式风格
  4. Option 类型 - 最优雅的方式

每种方式都有优缺点，选择哪种取决于具体的应用场景。

方式一：返回 null ​

基本思路 ​

当操作可能失败时，直接返回 null 来表示失败。这是最简单的错误处理方式。

代码示例 ​

typescript
function ask(): Date | null {
  return prompt('你的生日是什么时候？');
}

function parse(birthday: string): Date | null {
  let date = new Date(birthday);
  if (!isValid(date)) {
    return null;
  }
  return date;
}

使用方式 ​

typescript
let date = parse(ask());
if (date) {
  console.info('生日是', date.toISOString());
} else {
  console.error('错误: 某些原因导致解析日期失败');
}

优缺点分析 ​

优点：

  • 实现简单，轻量级
  • 用户输入无效时返回 null，类型系统会检查我们是否处理了这种情况

缺点：

  • 比较 null 的操作会让代码变得冗长
  • 需要在每个可能返回 null 的操作后都检查
  • 没有提供失败的具体原因信息

方式二：抛出异常 ​

基本思路 ​

当发生错误时，直接抛出异常，而不是返回 null。这样可以处理特定的失败类型，并获得一些失败的元数据。

代码示例 ​

typescript
function parse(birthday: string): Date {
  let date = new Date(birthday);
  if (!isValid(date)) {
    throw new RangeError('输入日期格式 YYYY/MM/DD');
  }
  return date;
}

使用方式 ​

typescript
try {
  let date = parse(ask());
  console.info('生日是', date.toISOString());
} catch (e) {
  if (e instanceof RangeError) {
    console.error(e.message);
  } else {
    // 处理其他异常
  }
}

注意事项 ​

  • 使用 try-catch 时要小心，不要意外吞掉所有可能的错误
  • 建议只捕获特定类型的异常，避免处理预期之外的错误

自定义错误类型 ​

为了更好地区分不同类型的错误，可以创建自定义错误类：

typescript
// 自定义错误类型
class InvalidDateFormatError extends RangeError {}
class DateIsInTheFutureError extends RangeError {}

function parse(birthday: string): Date {
  let date = new Date(birthday);
  if (!isValid(date)) {
    throw new InvalidDateFormatError('输入日期格式 YYYY/MM/DD');
  }
  if (date.getTime() > Date.now()) {
    throw new DateIsInTheFutureError('你是一个时间旅行者？');
  }
  return date;
}

处理多种异常类型 ​

typescript
try {
  let date = parse(ask());
  console.info('生日是', date.toISOString());
} catch (e) {
  if (e instanceof InvalidDateFormatError) {
    console.error(e.message);
  } else if (e instanceof DateIsInTheFutureError) {
    console.error(e.message);
  } else {
    throw e;  // 重新抛出未知错误
  }
}

进一步优化 ​

考虑到实际开发中，工程师可能懒得处理这么多异常类型，更可能写出这样的代码：

typescript
try {
  let result = parse(ask()) || parse(...)
  if (result instanceof InvalidDateFormatError) {
    // 处理错误
  }
} catch (e) {
  // 简单粗暴地处理所有错误
}

但这种做法不够优雅，因为它强制用户处理所有三种情况：成功、异常1、异常2。是否有更好的方法呢？

方式三：返回异常 ​

基本思路 ​

TypeScript 本身不支持 throws 声明，但我们可以通过联合类型来实现类似效果：

typescript
function parse(
  birthday: string
): Date | InvalidDateFormatError | DateIsInTheFutureError {
  let date = new Date(birthday);
  if (!isValid(date)) {
    return new InvalidDateFormatError('输入日期格式 YYYY/MM/DD');
  }
  if (date.getTime() > Date.now()) {
    return new DateIsInTheFutureError('你是一个时间旅行者？');
  }
  return date;
}

使用方式 ​

现在消费者必须处理所有三种情况：

typescript
let result = parse(ask());
if (result instanceof InvalidDateFormatError) {
  // 处理格式错误
  console.error(result.message);
} else if (result instanceof DateIsInTheFutureError) {
  // 处理未来日期错误
  console.error(result.message);
} else {
  // 成功情况
  console.info('生日是', result.toISOString());
}

优势分析 ​

这种方法成功利用了 TypeScript 的类型系统优势：

  • 编码时明确异常 - 在 parse 函数签名中明确了可能的异常
  • 告知消费者具体异常 - 调用者清楚知道可能抛出哪些异常
  • 强制处理所有异常 - 类型系统确保消费者处理（或重新抛出）每个异常

潜在问题 ​

一个懒惰的程序员可能会避免单独处理每个错误，而是这样写：

typescript
let result = parse(ask()); // 返回 Date 或者一个 Error
if (result instanceof Error) {
  console.error(result.message);
} else {
  console.info('生日是', result.toISOString());
}

当然，程序可能因为内存不足或栈溢出而崩溃，但这些是我们无法从中恢复的异常。

函数链式调用的复杂性 ​

这种方法有一个缺点：链式调用和嵌套错误操作会很快变得冗长。如果一个函数调用另一个可能失败的函数，那么该函数也需要明确处理错误：

typescript
function x(): T | Error1 {
  // ...
}

function y(): U | Error1 | Error2 {
  let a = x();
  if (a instanceof Error) {
    return a;
  }
  // 继续处理...
}

function z(): U | Error1 | Error2 | Error3 {
  let a = y();
  if (a instanceof Error) {
    return a;
  }
  // 返回结果
}

明确处理错误确实冗长，但这给了我们出色的安全性。

方式四：Option 类型 ​

基本概念 ​

现在我们可以通过使用特殊用途的数据类型来处理异常，这种方法有一些相比于处理值和错误的明显优势（类似性、与其他类型的互操作性等）。

Option 类型和 Either 类型在本章中，我们只讨论 Option 类型；另外两个类似的类型留给读者练习。

注意：Try、Option 和 Either 数据类型不会内置到 JavaScript 环境中，如：Node.js、Chrome、Safari、Edge 或 Firefox 等。如果你想在这些类型上使用它们，你需要自己实现或从 NPM 上安装自己写的。

Option 类型的来源 ​

Option 类型来自函数式编程语言如 Haskell、OCaml、Scala 和 Rust。其核心思想是，不返回一个值，也不抛出一个异常，而是返回一个容器，该容器可能包含也可能不包含一个值。容器有一些定义在其上的方法，让你可以链式操作，即使可能没有值在其中。

实现 Option 类型 ​

typescript
function parse(birthday: string): Option<Date> {
  let date = new Date(birthday);
  if (!isValid(date)) {
    return None;
  }
  return Some(date);
}

Option 类型的设计思路 ​

如之前所提到的，Option 是一个容器，要么包含某些东西，要么什么都不包含。让我们深入了解这意味着什么。

当 Option 真正发光时，是当你需要在一行中进行多个操作，其中每个操作都可能失败。

例如，假设我们总是假设解析成功，parse 可能会失败。但是如果我们很早就知道生日是无效的，我们不应该继续计算。我们可以用另一个 option 来建模：

typescript
function ask() {
  return prompt('你的生日是什么时候？')
}

ask()
  .map(parse)
  .map(date => date.toISOString())
  .forEach(date => console.info('生日是', date))
  // 错误："无效的 Property 'toISOString' 不存在类型 'Date[]'"
  // forEach date => console.info('生日是', date)

Option 类型的结构 ​

Option 是通过两个类实现的接口：Some 和 None（如图 7.1）。它们是 Option 的两种类型，Some 包含一个值，None 表示失败。

基本结构：

  • Option 接口定义了通用的方法
  • Some 类包含一个值，表示成功
  • None 类不包含值，表示失败

Option 的核心特点 ​

  • Option 是一个接口，Some 和 None 都实现这个接口
  • Option 是一个联合类型，表示要么是 Some 要么是 None

这种方法比起单纯地在所有地方比较 null 有几个好处：

  1. 类型安全 - 强制开发者处理可能的失败情况
  2. 链式操作 - 可以优雅地处理一系列可能失败的操作
  3. 语义清晰 - 明确表达了"可能有值，也可能没有值"的概念

Option 类型的具体实现 ​

基本概念 ​

  • Some 表示成功的操作，包含一个值。就像我们之前使用的数组一样，Some 是该值的容器
  • None 表示失败的操作，不包含任何值

这两种类型等价于我们基于数组的 Option 实现：

  • Option 相当于 [值] | []
  • Some 相当于 [值]
  • None 相当于 []

核心方法说明 ​

flatMap 方法：

  • 在 Some 上调用 flatMap 时，我们传入一个函数 f，flatMap 用 Some 的值调用 f
  • 由于 None 表示失败的计算，调用 flatMap 在 None 上总是返回 None
  • 一旦计算失败，我们就无法从中恢复

getOrElse 方法：

  • 调用 getOrElse 在 Some 上返回包含的值
  • 调用 getOrElse 在 None 上返回我们传递给 getOrElse 的默认值

使用示例和结果 ​

表 7.1：在 Some 和 None 上调用 flatMap() 的结果

也就是说，我们知道在 None 上映射总是得到 None，在 Some 上映射会得到 Some 或 None，这取决于我们调用的 f 函数。

Option 类型的完整实现 ​

我们已经推导出了一个功能完备、最小化的 Option 类型，让我们能够安全地执行可能失败的操作。我们像这样使用它：

基本使用示例：

javascript
let result = Option()
  .flatMap(ask)      // 获取用户输入
  .flatMap(parse)    // 解析日期
  .getOrElse('解析出错，某种原因')

// 或者获取具体值
ask()
  .flatMap(parse)
  .flatMap(date => Some(date.toISOString()))
  .getOrElse('错误：解析日期失败，某种原因')

Option 类型的工作原理 ​

基本工作流程：

  • 如果消费者调用 Option，我们返回一个 None
  • 否则，我们返回一个 Some，其中包含用户传入的值的类型
  • 最后，我们需要计算一个更精确的类型上界

类型安全的优势： Option 提供了强大的方式来处理一系列可能失败的操作。它们给你出色的类型安全，并向消费者发出信号，通过类型系统告知某个给定的操作可能会失败。

然而，Option 也有其缺点。它们表示失败时用 None，所以你不能获得 Option 失败时的具体信息（你有更明确的包装器类型来返回 Option，或者使用例外处理的替代方法）。

Option 类型的优缺点总结 ​

优点：

  • 类型安全 - 强制处理可能的失败情况
  • 链式操作 - 优雅地处理一系列可能失败的操作
  • 短路特性 - 一旦失败就停止后续操作
  • 函数式风格 - 避免了大量的 if-else 检查

缺点：

  • 缺乏具体错误信息 - 只知道失败了，不知道为什么失败
  • 学习成本 - 需要理解函数式编程概念
  • JavaScript 原生不支持 - 需要自己实现或引入第三方库

本章总结 ​

在这一章中，我们覆盖了在 TypeScript 中表示和处理错误的不同方式：

四种错误处理方式比较 ​

1. 返回 null

  • 是否要简单地发出信号表示某个失败了（null，Option），或者获得更多关于为什么失败的信息（返回异常，或抛出异常）

2. 抛出异常

  • 是否需要强制消费者显式处理每个可能的异常（返回异常），或者进行更易于处理的处理当它们出现时（抛出异常）

3. 返回异常

  • 返回异常和 Option 各有优缺点，选择取决于具体应用场景

4. Option 类型

  • 最优雅的函数式解决方案，但需要一定的学习成本

选择建议 ​

选择哪种机制取决于你的应用。当我涵盖每种错误处理机制时，我已经谈论了它们的优缺点，这样你就可以为自己做出正确的选择。

练习题 ​

练习1：API 错误处理设计 ​

为以下 API 设计一种错误处理方式，使用本章中的任一模式。在这个 API 中，每个操作都可能失败——可以自由更新 API 的方法签名以允许失败（或者不更新，如果你更喜欢）。思考如何在处理错误的同时执行一系列操作（例如：获取登录用户的 ID，然后获取他们的好友列表，然后获取每个好友的姓名）：

javascript
class API {
  getLoggedInUserID() // 返回 UserID
  getFriendIDs(userID) // 返回 UserID[]
  getUserName(userID) // 返回 string
}

练习要点：

  • 选择合适的错误处理模式
  • 考虑链式操作的错误传播
  • 思考如何优雅地处理多个可能失败的步骤

补充说明 ​

  1. Java 的 throws 子句：如果你之前没有使用过 Java，throws 子句用来指示方法可能抛出哪些类型的运行时异常，这样消费者就必须处理这些异常。

  2. Option 类型的别名：Option 类型也被称为 Maybe 类型。

  3. 扩展学习：可以搜索"try type"或"either type"来了解更多相关类型的信息。

练习思路提示 ​

这个练习的核心是要理解：

  • 如何设计能够优雅处理失败的 API
  • 如何在多个可能失败的操作之间传递错误
  • 不同错误处理模式在实际应用中的选择考量

你可以尝试用四种不同的方式来实现这个 API，比较它们的优缺点。

练习题答案 ​

方案一：返回 null ​

javascript
class API {
  getLoggedInUserID() {
    // 模拟可能失败的登录检查
    const isLoggedIn = Math.random() > 0.1;
    return isLoggedIn ? 'user123' : null;
  }
  
  getFriendIDs(userID) {
    if (!userID) return null;
    // 模拟网络请求可能失败
    const success = Math.random() > 0.2;
    return success ? ['friend1', 'friend2', 'friend3'] : null;
  }
  
  getUserName(userID) {
    if (!userID) return null;
    const success = Math.random() > 0.1;
    return success ? `用户${userID}的名字` : null;
  }
}

// 使用方式
function getFriendsNames() {
  const api = new API();
  
  const userID = api.getLoggedInUserID();
  if (!userID) {
    console.log('用户未登录');
    return;
  }
  
  const friendIDs = api.getFriendIDs(userID);
  if (!friendIDs) {
    console.log('获取好友列表失败');
    return;
  }
  
  const friendNames = [];
  for (const friendID of friendIDs) {
    const name = api.getUserName(friendID);
    if (name) {
      friendNames.push(name);
    }
  }
  
  console.log('好友姓名:', friendNames);
}

方案二：抛出异常 ​

javascript
class UserNotLoggedInError extends Error {}
class NetworkError extends Error {}
class UserNotFoundError extends Error {}

class API {
  getLoggedInUserID() {
    const isLoggedIn = Math.random() > 0.1;
    if (!isLoggedIn) {
      throw new UserNotLoggedInError('用户未登录');
    }
    return 'user123';
  }
  
  getFriendIDs(userID) {
    const success = Math.random() > 0.2;
    if (!success) {
      throw new NetworkError('网络请求失败');
    }
    return ['friend1', 'friend2', 'friend3'];
  }
  
  getUserName(userID) {
    const success = Math.random() > 0.1;
    if (!success) {
      throw new UserNotFoundError(`用户 ${userID} 不存在`);
    }
    return `用户${userID}的名字`;
  }
}

// 使用方式
function getFriendsNames() {
  const api = new API();
  
  try {
    const userID = api.getLoggedInUserID();
    const friendIDs = api.getFriendIDs(userID);
    
    const friendNames = [];
    for (const friendID of friendIDs) {
      try {
        const name = api.getUserName(friendID);
        friendNames.push(name);
      } catch (e) {
        if (e instanceof UserNotFoundError) {
          console.log(`跳过不存在的用户: ${friendID}`);
        } else {
          throw e; // 重新抛出其他异常
        }
      }
    }
    
    console.log('好友姓名:', friendNames);
  } catch (e) {
    if (e instanceof UserNotLoggedInError) {
      console.log('请先登录');
    } else if (e instanceof NetworkError) {
      console.log('网络连接失败，请稍后重试');
    } else {
      console.log('未知错误:', e.message);
    }
  }
}

方案四：Option 类型 ​

javascript
// 简化的 Option 实现
class Some {
  constructor(value) {
    this.value = value;
  }
  
  flatMap(fn) {
    return fn(this.value);
  }
  
  getOrElse(defaultValue) {
    return this.value;
  }
  
  isSome() {
    return true;
  }
}

class None {
  flatMap(fn) {
    return new None();
  }
  
  getOrElse(defaultValue) {
    return defaultValue;
  }
  
  isSome() {
    return false;
  }
}

class API {
  getLoggedInUserID() {
    const isLoggedIn = Math.random() > 0.1;
    return isLoggedIn ? new Some('user123') : new None();
  }
  
  getFriendIDs(userID) {
    const success = Math.random() > 0.2;
    return success ? new Some(['friend1', 'friend2', 'friend3']) : new None();
  }
  
  getUserName(userID) {
    const success = Math.random() > 0.1;
    return success ? new Some(`用户${userID}的名字`) : new None();
  }
}

// 使用方式
function getFriendsNames() {
  const api = new API();
  
  const result = api.getLoggedInUserID()
    .flatMap(userID => api.getFriendIDs(userID))
    .flatMap(friendIDs => {
      const names = friendIDs
        .map(id => api.getUserName(id))
        .filter(option => option.isSome())
        .map(option => option.getOrElse(''));
      
      return names.length > 0 ? new Some(names) : new None();
    });
  
  const friendNames = result.getOrElse([]);
  
  if (friendNames.length > 0) {
    console.log('好友姓名:', friendNames);
  } else {
    console.log('无法获取好友信息');
  }
}

方案对比总结 ​

方案优点缺点适用场景
返回 null简单直观，性能好缺乏错误信息，需要大量检查简单应用，错误类型少
抛出异常错误信息丰富，代码简洁控制流复杂，容易遗漏处理传统应用，异常情况确实异常
返回异常强制错误处理，类型安全代码冗长，学习成本高高可靠性要求的应用
Option 类型函数式，链式操作优雅学习成本高，调试困难函数式编程项目

推荐选择：

  • 小型项目：返回 null
  • 传统 Web 应用：抛出异常
  • 企业级应用：返回异常
  • 函数式项目：Option 类型
