this 和对象原型 ​

本章聚焦两块：

  • this 的绑定规则与误区
  • 对象原型、原型链与委托

一、this 绑定五原则 ​

  1. 默认绑定（非严格模式下指向全局对象，严格模式下为 undefined）
  2. 隐式绑定（由调用点的宿主对象决定）
  3. 显式绑定（call/apply/bind）
  4. new 绑定（构造调用）
  5. 箭头函数（词法绑定，取决于定义处的 this）

优先级：new > 显式 > 隐式 > 默认；箭头函数独立于此规则，始终词法捕获。

js
function foo() { console.log(this.a) }
const obj = { a: 1, foo }
const a = 2
foo()        // 默认绑定：严格模式下 undefined；非严格 2
obj.foo()    // 隐式绑定：1
foo.call({ a: 3 }) // 显式绑定：3
new foo()    // new 绑定：this 指向新对象

二、常见陷阱 ​

  • 丢失隐式绑定：解构/赋值传递函数引用导致 this 丢失
  • 回调中的 this：事件/定时器需要显式 bind 或用箭头函数
  • DOM/框架包装：实际调用点变化影响 this

实例：丢失绑定与解决 ​

js
function show(){ console.log(this.id) }
const obj = { id: 'OBJ', show }

// 1) 直接调用：默认绑定
show() // 严格模式 undefined；非严格 window.id

// 2) 隐式绑定
obj.show() // 'OBJ'

// 3) 丢失绑定：解构/赋值
const fn = obj.show
fn() // 严格模式 undefined

// 解决：显式绑定或硬绑定
const bound = obj.show.bind(obj)
bound() // 'OBJ'

### 原型委托 vs class 继承（对照）

```js
// 原型委托
const Animal = { speak(){ return `${this.name} makes a sound` } }
const dog = Object.create(Animal)
dog.name = 'Dog'
dog.speak() // "Dog makes a sound"

// class 语法糖（运行时依然是原型）
class AnimalC { constructor(name){ this.name = name } speak(){ return `${this.name} makes a sound` } }
class DogC extends AnimalC { speak(){ return `${super.speak()}!` } }
new DogC('Dog').speak()

提示：TS 编译后的 class/extends 依旧透过 __extends 等辅助实现原型链连接。

new 与显式绑定的优先级 ​

js
function Foo(){ this.x = 1 }
const bar = Foo.bind({ x: 0 })
const inst = new bar()
console.log(inst.x) // 1 —— new 绑定优先，忽略 bind 的 this

三、对象原型与委托 ​

  • [[Prototype]] 内部槽（通常通过 __proto__ 或 Object.getPrototypeOf 访问）
  • 原型链是“查找失败 -> 委托给原型”的动态过程
  • 与“类继承”不同：JS 采用行为委托而非复制
js
const A = { hi() { return 'A' } }
const B = Object.create(A)
B.hi()           // 'A' —— 委托到原型 A
B.hi = () => 'B' // 覆盖
B.hi()           // 'B'

四、与 TypeScript 的衔接 ​

  • TS 的类在编译后仍基于原型；理解委托有助于读懂编译产物
  • interface/类型别名只在编译期存在，不影响运行时原型关系
  • 用 TS 声明构造函数、实例、原型成员的类型，更易发现 API 设计问题

练习 ​

  • 用不同调用位置推导 this 结果，并解释原因
  • 编写一个基于 Object.create 的“委托链”，体会与 class 继承的差异
  • 用 TS 为构造函数和实例方法标注类型，检查 this 使用是否正确

练习参考答案 ​

  1. 调用位置决定 this（推导并解释）
js
function who(){ return this.tag }
const a = { tag: 'A', who }
const b = { tag: 'B' }

who()            // 严格模式 undefined；非严格 window.tag（默认绑定）
a.who()          // 'A'（隐式绑定）
a.who.call(b)    // 'B'（显式绑定）
const f = a.who
f()              // 严格模式 undefined（丢失绑定）
const g = a.who.bind(a)
g()              // 'A'（硬绑定）

解释：根据“调用点”套用优先级：new > 显式 > 隐式 > 默认；bind 创建新函数并固定 this。

  1. 基于 Object.create 的委托链 vs class 继承
js
// 委托链：按需覆盖，查找失败时向上委托
const A = { say(){ return `A` } }
const B = Object.create(A)
B.say = () => `B` // 覆盖
const C = Object.create(B)
C.say() // 'B'，C 无 say -> 委托 B

// class 继承：方法复制到原型链上（运行时依然走原型查找）
class A2 { say(){ return 'A2' } }
class B2 extends A2 { say(){ return 'B2' } }
class C2 extends B2 {}
new C2().say() // 'B2'

结论：两者运行时都依赖原型链；委托强调“按需覆盖与查找”，class 强调“类型层级与语义组织”。

  1. TS 中为构造函数与实例方法标注 this
ts
class Counter {
  private count = 0
  inc(this: Counter){ this.count++ }
  value(){ return this.count }
}

const c = new Counter()
const inc = c.inc
// inc() // TS 报错：缺少 this 上下文
const incBound = c.inc.bind(c)
incBound()

说明：通过 this 参数标注，避免丢失绑定；或在使用处用 bind 硬绑定。

进阶挑战（含答案） ​

  1. 软绑定（softBind）思路实现
js
function softBind(fn, defaultThis){
  return function(...args){
    const ctx = (this && this !== globalThis) ? this : defaultThis
    return fn.apply(ctx, args)
  }
}
function who(){ return this.tag }
const a = { tag: 'A' }
const b = { tag: 'B' }
const g = softBind(who, a)
console.log(g())            // 'A'
console.log(g.call(b))      // 'B'

答案：软绑定在默认情况下使用提供的 this，但允许显式绑定覆盖。

  1. 箭头函数捕获 this 与对象方法
js
const obj = {
  id: 1,
  inc(){ setTimeout(() => { this.id++ }, 0) },
  bad(){ setTimeout(function(){ this.id++ }, 0) }
}
obj.inc()
setTimeout(() => console.log('inc id =', obj.id), 5)
obj.bad()
setTimeout(() => console.log('bad id =', obj.id), 5)

答案：inc 使用箭头函数词法绑定，正确递增；bad 中的普通函数默认绑定到全局/undefined，导致不按预期。
