作用域和闭包（Scope & Closures） ​

一、作用域的形成 ​

  • 词法作用域：由代码书写位置决定
  • 变量解析顺序：当前作用域 -> 外层作用域 -> 全局
  • 编译阶段建立环境记录（Environment Record）
js
function outer(){
  const x = 1
  function inner(){ console.log(x) }
  return inner
}
const f = outer()
f() // 1 —— 闭包保留了对 x 的访问

二、闭包的本质 ​

  • “函数 + 其定义时的词法环境”的组合
  • 不是“把变量复制一份”，而是持有对环境的引用

三、常见用法与陷阱 ​

  • 循环中的闭包与 let/var 的差异
  • 惰性函数、缓存、工厂函数
js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0) // 3 3 3
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0) // 0 1 2
}

四、与 TypeScript 的衔接 ​

  • TS 的类型不改变闭包语义，但能限制外部变量的可变性（如 readonly）
  • 在函数式风格下配合类型参数（泛型）表达数据流更清晰

练习 ​

  • 写出 2 种创建闭包的方式，并分析内存释放时机
  • 将 var 循环闭包改写为立即调用函数表达式（IIFE）或 let

练习参考答案 ​

  1. 写出 2 种创建闭包的方式，并分析内存释放时机
js
// 方式A：返回内部函数
function outer(){
  const cache = { v: 1 }
  return function inner(){
    return cache.v
  }
}
const f = outer()
f() // 1
// 释放时机：f 置为 null 或超出可达范围，且无其他引用指向 inner 或 cache

// 方式B：事件处理器（闭包引用了外部变量）
function attach(){
  const msg = 'hello'
  const handler = () => console.log(msg)
  document.body.addEventListener('click', handler)
  return () => document.body.removeEventListener('click', handler)
}
const dispose = attach()
// 释放时机：调用 dispose() 解除引用；否则闭包长期存活可能造成泄漏

要点：闭包并非问题，关键在于生命周期管理；及时解除对闭包或其捕获环境的最后引用。

  1. 将 var 循环闭包改写为 IIFE 或 let
js
// 原问题
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0) // 3 3 3
}

// 解法A：let（推荐）
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0) // 0 1 2
}

// 解法B：IIFE 捕获当次 i
for (var i = 0; i < 3; i++) {
  ((i) => setTimeout(() => console.log(i), 0))(i)
}

解释：let 在每次迭代创建新绑定；IIFE 用参数将当次值封进函数作用域，二者都能避免共享同一 var 绑定。

进阶挑战（含答案） ​

  1. 参数默认值与 TDZ
js
function foo(a = b, b = 1){
  console.log(a, b)
}
// foo() // ?

答案：调用会抛 ReferenceError。原因：评估 a 的默认值时需要读取 b，但 b 仍处于其参数作用域的 TDZ。

  1. 闭包读取“最新值”与快照
js
let x = 0
const fns = []
for (let i = 0; i < 2; i++) {
  fns.push(() => console.log(x))
}
x = 42
fns[0]() // ?

答案：42。闭包捕获的是对 x 的引用而非拷贝。若要固定当时的值，可快照：

js
for (let i = 0; i < 2; i++) {
  const v = x
  fns.push(() => console.log(v))
}

五、深入要点补充 ​

1) 暂时性死区（TDZ）与提升 ​

  • var 声明会“提升”（hoist），在作用域顶部创建绑定，未赋值前为 undefined
  • let/const 在作用域开始到实际声明之间处于 TDZ，访问将抛 ReferenceError
js
console.log(a) // undefined（var 提升）
var a = 1

console.log(b) // ReferenceError（TDZ）
let b = 2

要点：TDZ 不是“未定义”，而是禁止访问；有助于暴露潜在顺序错误。

2) IIFE 与块级作用域 ​

  • 经典：IIFE 隔离变量，避免泄漏到外层
js
(function(){
  const local = 1
  // …
})()
  • 现代：使用块级作用域即可达到隔离效果
js
{
  const local = 1
}

3) 循环与闭包的三种可靠写法 ​

js
// A: let 形成每次迭代的新绑定
for (let i = 0; i < 3; i++) setTimeout(() => console.log(i), 0)

// B: IIFE 捕获当次 i 的值
for (var i = 0; i < 3; i++) (i => setTimeout(() => console.log(i), 0))(i)

// C: forEach 的回调形参形成新的词法绑定
[0,1,2].forEach(i => setTimeout(() => console.log(i), 0))

4) 词法屏蔽与名称冲突 ​

  • 内层同名变量会屏蔽外层绑定；避免随意复用名称
  • 建议：较短作用域使用短名，跨作用域共享使用具名/前缀名

六、内存与生命周期 ​

  • 闭包会使其捕获的环境保持可达，从而阻止垃圾回收
  • 释放条件：不再有可达引用指向闭包（函数或其返回对象）
  • 常见内存泄漏场景：
    • 将闭包挂到全局或长生命周期单例上却未清理
    • DOM 事件处理器中的闭包未移除
  • 建议：
    • 组件卸载/作用域结束时显式清理引用（如 removeEventListener）
    • 避免将大型对象闭包化；按需捕获（解构挑选所需字段）
    • 使用浏览器 DevTools 的 Memory/Profile 工具观测闭包保留的变量

七、实践清单（Checklist） ​

  • 需要“每次迭代独立值”时优先用 let 或 IIFE
  • 需要隔离变量时优先用块级作用域而非滥用 IIFE
  • 避免在闭包中捕获“可变且广域”的对象引用
  • 对事件/回调注册建立对称的清理流程
  • 在 TS 中利用 readonly/类型缩窄限制可变性，但记住它不改变闭包行为
